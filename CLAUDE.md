## Правило: изменения index.html

Любой запрос на изменение **структуры или визуала сайта** (`index.html`) — новые блоки, вкладки, дизайн, компоненты — требует предварительного чтения алгоритма сборки:

→ [docs/spec-pilot.md](docs/spec-pilot.md)

**Когда читать:** запрос содержит «добавь», «сделай», «перенеси», «измени» применительно к `index.html` или к тому как сайт выглядит или работает.

**Когда не читать:** правка сигналов (`signals.json`, `SIGNALS.md`), обновление документации, мелкие однострочные правки с очевидным решением.

**Исключение — новый кластер.** Создание нового `cluster` (Шаг 4) требует точечных записей в **двух** отдельных lookup-таблицах внутри `index.html` — `DIGEST_CLUSTER_LABELS` (короткие подписи, фильтры вкладки Сигналы) и `CLUSTER_LABELS` (длинные подписи, карточки «Главные нарративы», отдельная функция) — иначе в соответствующем месте сайта вместо человеческой подписи покажется сырой `SNAKE_CASE`. Это точечная запись в обе lookup-таблицы (`new_cluster_key: 'ЭМОДЗИ ПОДПИСЬ'`), не структурное и не визуальное изменение — `spec-pilot.md` для неё не читать. Всё остальное в `index.html` по-прежнему не трогать при работе с сигналами.

## Сигнал: схема объекта

Каждый новый сигнал — **два файла одновременно**: `signals.json` + `SIGNALS.md`.

```json
{
  "id": "STR-2026-0623-001",
  "date": "YYYY-MM-DD",
  "cat": "ownership",
  "catLabel": "🏛️ ВЛАДЕНИЕ",
  "dir": "neg",
  "horizon": "mid",
  "theme": "institutionalization",
  "weight": "media",
  "actor": "corporate",
  "flow": "outflow",
  "signal": "Заголовок сигнала",
  "data": ["метрика 1", "метрика 2"],
  "context": "Исторический контекст и аналогии.",
  "caveat": "Риски и ограничения интерпретации.",
  "alternatives_considered": [
    "Отклонённая интерпретация — почему отклонена"
  ],
  "alternative_scenario": "Опционально, если уверенность не высокая: связный альтернативный путь развития, если основная интерпретация неверна",
  "facts": [
    { "key": "entity_id.metric", "value": 123456, "unit": "BTC", "as_of": "YYYY-MM-DD" }
  ],
  "source": "Источник (месяц год)",
  "links": {
    "confirms": ["id сигнала который этот усиливает"],
    "contradicts": ["id сигнала с которым в противоречии"],
    "context_chain": ["id сигнала из контекста которого вырастает этот"]
  },
  "narrative_role": "complication",
  "cluster": "strategy_model_stress",
  "tension": "краткое описание противоречия которое сигнал раскрывает",
  "macro_implication": "что означает на уровне общей картины"
}
```

- Все три поля `links` — массивы **id** (не заголовков)
- `index.html` для сигналов не трогать — данные загружаются из `signals.json` через fetch

## Справочник значений полей

> Формат и допустимые значения — здесь. Доменный смысл значений (связь с онтологией) → [docs/DATA_DICTIONARY.md](docs/DATA_DICTIONARY.md).

**`id`** — формат `PREFIX-YYYY-MMDD-NNN`. Префикс выбирается по `theme` **на момент создания**:

| Префикс | theme | Смысл темы |
|---------|-------|-----------|
| `SUP` | `supply` | Дефицит, эмиссия, потерянные монеты |
| `STR` | `institutionalization` | ETF, корпорации, правительства, владение |
| `INF` | `infrastructure` | Lightning, Layer 2, майнинг |
| `MAC` | `macro` | Ставки, ликвидность, макро-давление |
| `NAR` | `narrative` | Философия, регуляция, нарративы |

**Правило неизменности `id`:** после записи `id` не меняется никогда — на него ссылаются `links` других сигналов. Если `theme` сигнала уточнилась позже, префикс остаётся историческим; расхождение префикса и темы задним числом не исправляется.

**`dir`** — что сигнал говорит о силе Bitcoin как актива в своём горизонте:

| Значение | Смысл |
|----------|-------|
| `pos` | усиливает позицию BTC |
| `neg` | создаёт давление на BTC |
| `neu` | структурный факт без направления |

**`horizon`** — временной горизонт:

| Значение | Смысл |
|----------|-------|
| `short` | недели–месяцы |
| `mid` | месяцы–год |
| `long` | годы, структурные изменения |

**`weight`** — достоверность источника:

| Значение | Смысл |
|----------|-------|
| `onchain` | верифицируемые данные блокчейна |
| `market` | биржевые данные, цена, объёмы |
| `media` | медиа, аналитики, оценки |
| `primary` | первичный источник (высказывание, документ) |

**`actor`** — субъект сигнала:

| Значение | Смысл |
|----------|-------|
| `etf` | биржевые фонды (BlackRock, Fidelity) |
| `corporate` | публичные компании (Strategy, MSTR) |
| `government` | государственные резервы |
| `defi` | протоколы, on-chain структуры |
| `retail` | розничные инвесторы |
| `miner` | майнеры |

**`flow`** — направление капитала:

| Значение | Смысл |
|----------|-------|
| `inflow` | капитал входит в BTC |
| `outflow` | капитал уходит из BTC |
| `internal` | перераспределение внутри структуры |
| `neutral` | движения капитала нет или неизвестно |

**`cat` / `catLabel`** — категория и её отображаемая метка. `catLabel` выводится из `cat` строго по таблице — никаких вариаций регистра, эмодзи или синонимов:

| cat | catLabel (канон) | Примеры |
|-----|------------------|---------|
| `onchain` | 📊 ON-CHAIN | Production Cost, MVRV, NVT |
| `macro` | 🏦 МАКРО | Ставки, ликвидность, DXY |
| `mining` | ⛏️ МАЙНИНГ | Хешрейт, сложность, капитуляция |
| `narrative` | 📰 НАРРАТИВ | Регуляция, институционалы, события |
| `layer2` | 🔗 LAYER 2 | Stacks, Lightning, сайдчейны |
| `ownership` | 🏛️ ВЛАДЕНИЕ | Распределение по типам владельцев |

> **Факт о рендере (2026-07-03):** в текущем `index.html` `catLabel` используется в разметке только как fallback, который не срабатывает — реальную подпись на карточке даёт `DIGEST_CLUSTER_LABELS[cluster]` (см. «Кластеры» ниже). `catLabel` тем не менее остаётся обязательным и канонизированным полем данных: это дата-контракт для будущих потребителей `signals.json` (не только текущий index.html), а чеклист Шага 8 по-прежнему требует его сверки с `cat`.

> Категория `ta` (технический анализ) депрецирована в v6.0: технический анализ выведен за границы предметной области (обоснование → [docs/BDKS.md](docs/BDKS.md), Раздел 1). Исторических сигналов с этой категорией нет; новые не создаются.

- Создание кластера = **четыре места одновременно** (не два — расхождение обнаружено и исправлено 2026-07-07 при создании `leverage_deleveraging_cycle`):
  1. `cluster` в новом сигнале (`signals.json`)
  2. `ontology.json` → `clusters` — фактический источник истины, на который явно ссылается описание поля `cluster` в `schemas/signal/v1.json` («должен существовать в ontology.clusters»); не проверяется тестами автоматически, но обязателен для непротиворечивости
  3. `DIGEST_CLUSTER_LABELS` (`index.html`, короткие подписи для фильтров вкладки Сигналы)
  4. `CLUSTER_LABELS` (`index.html`, отдельный словарь — длинные подписи для карточек «Главные нарративы»; ранее в этой инструкции не упоминался вообще)

  Пропуск любого из 2–4 — кластер существует в данных, но на сайте отображается некрасиво (сырой `SNAKE_CASE`) в соответствующем месте. Для этой правки (все четыре — lookup-таблицы, не структура/визуал) `spec-pilot.md` по-прежнему не читать, см. «Правило: изменения index.html» выше.

## Алгоритм обработки сигнала

**Правило видимости рассуждения:**
Шаги 3 (Разбор), 4 (Кластеризация), 5 (Связывание) и 6 (Синтез) содержат интерпретационные решения, не механические проверки: выбор `narrative_role`, формулировка `tension`/`macro_implication`, состав `confirms`/`contradicts`/`context_chain`, причина выбора `cluster`. Эти решения и их обоснование показываются пользователю в чате до сборки итогового JSON — отдельным разбором по шагам, а не только результатом.

«Сводка после записи сигнала» (см. ниже) — краткое резюме уже показанных и принятых решений, а не единственное место, где они становятся видны. Свернуть весь процесс сразу в эту сводку недопустимо, даже если шаблон сводки кажется самодостаточным.

```
✗ Claude молча проходит Шаги 3–6 внутри себя, показывает только готовый JSON + сводку
✓ Claude показывает разбор (что это / контекст / что значит / оговорки), затем
   обоснование cluster и links, и только потом — оформление и сводку
```

### Шаг 1 — Приём материала

Определить формат входящего материала: текст, ссылка, скриншот.
Зафиксировать что именно прислал пользователь — это исходная точка, не интерпретация.

**Ограничение:** Claude не наполняет сигналы из собственных знаний. Только из материала пользователя. Переход к следующему сигналу — только после явного подтверждения пользователя.

### Шаг 2 — Исследование

Найти первоисточник:
- Если дана ссылка → `fetch` по ней
- Если нет → `web_search` по ключевым фактам материала

Найти 2–3 дополнительных источника по типу сигнала:

| Тип сигнала | Источники |
|------------|-----------|
| Корпоративные покупки BTC | Arkham, SEC filing, пресс-релиз |
| ETF / регуляторика | SEC.gov, Bloomberg ETF, проспект |
| Майнинг / инфраструктура | Hashrate Index, Glassnode, отчёты |
| Макро / on-chain | Glassnode, CryptoQuant, VanEck ChainCheck |
| Нарративы / медиа | Bitcoin Magazine, CoinDesk, The Block |

- Если источники противоречат — зафиксировать расхождение, оно войдёт в `caveat`
- Если первоисточник недоступен — сообщить пользователю и работать с тем что есть

### Шаг 3 — Разбор

Структурировать собранный материал по четырём пунктам:

- **Что это** — краткое определение события или факта
- **Контекст** — исторические аналогии, предыдущие прецеденты
- **Что означает сейчас** — интерпретация для BTC
- **Важные оговорки** — риски, ограничения, противоречия источников

Внутри пункта «Что означает сейчас» — рассмотреть минимум одну содержательную альтернативную интерпретацию и явно её отклонить с причиной (не соломенное чучело — альтернатива должна объяснять те же факты иначе). Отклонённые альтернативы фиксируются в `alternatives_considered`, не только проговариваются в чате.

Если уверенность в выбранной интерпретации не высокая (качественная оценка — источник не первичный, факты неполные, есть заметное расхождение источников) — сформулировать `alternative_scenario`: связный путь развития, которым может пойти ситуация, если основная интерпретация неверна, не просто «а вдруг наоборот» (ADR-015). Поле опционально — не заполнять, если уверенность высокая.

Это читаемый промежуточный артефакт. Из него питаются поля JSON:

| Пункт разбора | Поля JSON |
|--------------|-----------|
| Что это | `signal`, `cat`, `catLabel`, `actor`, `flow` |
| Контекст | `context` |
| Что означает сейчас | `dir`, `horizon`, `weight`, `theme`, `alternatives_considered`, `alternative_scenario` |
| Важные оговорки | `caveat`, `data` |

Поля `id`, `date`, `source` заполняются отдельно: `source` — название источника и месяц, `id` — на шаге 8.

**Правило определения `date`:**
`date` — дата события, берётся из самого проверенного источника, найденного на Шаге 2, а не выводится расчётом от относительных формулировок пользователя («вчера», «сегодня», «на этой неделе») и не от даты сессии.

Приоритет источника даты:
1. Дата/период в самом первоисточнике (официальный релиз, regulatory filing, документ) — высший приоритет
2. Дата публикации наиболее авторитетного вторичного источника (Шаг 2, колонка «Источники»)
3. Если ни один источник дату не называет явно — спросить пользователя напрямую, не предполагать

```
✗ Пользователь: «отчиталась вчера» → Claude молча вычисляет: сегодня минус 1 день
✓ Claude берёт дату из самого filing/источника; если расходится с «вчера» пользователя —
   фиксирует это расхождение и уточняет, а не выбирает дату самостоятельно
```

Если источники расходятся в дате — расхождение идёт в `caveat`, используется дата самого первичного источника.

### Шаг 4 — Кластеризация

Определить `theme` по справочнику полей.

Определить `cluster` по двум признакам одновременно:
1. `theme` совпадает с темой кластера
2. Субъект события (`actor` + суть действия) совпадает с описанием кластера

Если оба признака совпадают → кластер найден.
Если только `theme` совпадает → потенциально новый кластер (создаётся при 2+ сигналах с общим противоречием которого нет в существующих).

Кластер назначается здесь. `tension` формулируется только на шаге 6 — не использовать его как критерий поиска на этом шаге.

### Шаг 5 — Связывание

Найти кандидатов: все сигналы с тем же `theme`. Если кластер определён — добавить все сигналы того же кластера.

Для каждого кандидата проверить три критерия:

**`confirms`** — оба сигнала говорят одно и то же но через разные механизмы?
→ Если да: добавить в `confirms`

**`contradicts`** — `macro_implication` двух сигналов описывают **несовместимые состояния одной системы**?
Тест: можно ли одновременно быть правым и в A, и в B?
→ Нет → добавить в `contradicts`
→ Да → не добавлять

**Запрещено** добавлять `contradicts` чтобы сделать сигнал победителем tension кластера. Чтобы сменить tension кластера — добавить **новый** сигнал с более сильным `tension` (см. «Нарративный синтез»); `tension` уже записанного и учтённого в синтезе сигнала не редактируется задним числом (Immutability Policy, `docs/NIES.md` AD-7, закрыт в пользу D13 — 2026-07-05).

**`context_chain`** — без понимания сигнала X текущий сигнал теряет смысл?
→ Если да: добавить в `context_chain`

Пустое поле — норма если после проверки всех кандидатов критерий не выполнен ни разу.
Не добавлять связь только потому что сигналы близки по теме.

### Шаг 6 — Синтез

Выполняется только после завершения шага 5.

**`tension`** — противоречие между двумя силами, не описание факта:
- Формула: «X vs Y» / «X несмотря на Y» / «X при условии что Y»
- Два конкретных механизма в столкновении
- Всегда начинается с заглавной буквы

| ❌ Плохо | ✅ Хорошо |
|---------|---------|
| «Strategy продолжает покупать BTC» (нет противоречия) | «Strategy наращивает долг для покупки BTC vs рынок ставит NAV-дисконт 0.83x» |
| «ETF показал отток» | «ETF-оттоки как поверхностное давление vs долгосрочные держатели покупают в 10x больше» |

**`macro_implication`** — ответить на вопрос «что это меняет для BTC структурно».
Не пересказ события — вывод об изменении в системе.

| ❌ Плохо | ✅ Хорошо |
|---------|---------|
| «Franklin подала заявку на ETF» | «Пассивный дивидендный поток как источник BTC-спроса — новая категория не зависящая от настроений рынка» |
| «Metaplanet купила Siiibo» | «BTC-казначейство эволюционирует от пассивного баланса к операционному движку: коллатерал → лицензия → продукты → комиссионный доход» |

**`narrative_role`** — выбрать одну роль:

| Роль | Критерий | Расшифровка для сводки |
|------|---------|------------------------|
| `trigger` | Первый сигнал нового кластера или резкий перелом | открывает цепочку |
| `complication` | Добавляет противоречие в существующий нарратив | усложняет нарратив |
| `resolution` | Закрывает `tension` предыдущих сигналов | закрывает противоречие |
| `background` | Структурный контекст без острого события | структурный контекст |

### Шаг 7 — Проверка

Сигнал готов к оформлению только если:

- `tension` сформулирован по формуле (не описание факта)
- `macro_implication` отвечает на «что меняется структурно» (не пересказ)
- `narrative_role` выбрана и обоснована
- `links` — все три поля проверены по критериям шага 5
- `alternatives_considered` содержит ≥1 содержательную альтернативу с причиной отклонения
- `alternative_scenario` заполнен, если уверенность не высокая (иначе — обоснованно пропущен, не забыт)

Если что-то не выполнено — вернуться к соответствующему шагу.

### Шаг 8 — Оформление

**Присвоить `id`:**
Формат `PREFIX-YYYY-MMDD-NNN`. Перед записью открыть `signals.json` и найти все существующие `id` с той же датой и тем же префиксом. `NNN` — следующий порядковый номер. Если `STR-2026-0626-001` уже есть → `STR-2026-0626-002`.

**Чеклист перед записью:**
- JSON валиден — нет незакрытых скобок, нет лишних запятых
- `id` уникален в `signals.json`
- `catLabel` соответствует `cat` по канонической таблице справочника
- Все обязательные поля заполнены (прошёл шаг 7)
- `ENTITIES.json` обновлён, если в сигнале участвуют значимые сущности (см. «База артефактов»)
- Если Шаг 4 завёл **новый** `cluster` — добавлена строка в `DIGEST_CLUSTER_LABELS` (`index.html`); для существующего кластера этот пункт не применим
- `signals.json` и `SIGNALS.md` обновляются в одном коммите
- Коммит использует префикс `signal:` (не `feat:`/`fix:`), формат —
  см. [`CONTRIBUTING.md`](CONTRIBUTING.md#commit-message-convention)

`index.html` не трогать — кроме одной строки в `DIGEST_CLUSTER_LABELS` при новом кластере (см. выше и чеклист).

## Нарративный синтез (кратко)

Полная документация → [ALGORITHM.md](docs/ALGORITHM.md). Карточка нарратива строится из полей сигналов — алгоритм не генерирует текст, он **выбирает лучшее** из написанного.

**Score сигнала** (формула `_score_signal`, ALGORITHM.md):

| Компонент | Значения |
|-----------|----------|
| Свежесть | ≤7 дней = **+3** · ≤30 дней = **+1** · старше = 0 |
| `weight` | `onchain` = **+4** · `primary` = **+3** · `market` = **+2** · `media` = **+1** |
| `narrative_role` | `trigger` = **+4** · `complication` = **+3** · `resolution` = **+2** · `background` = 0 |
| `links.contradicts` | **+5 за каждый id** в списке (без верхнего предела) |

> `tension` в score не участвует — он определяет текст карточки, не ранжирование.

**Что определяет текст карточки:**

```
tension          → главный крючок (золотая полоса) — из сигнала с max contradicts
macro_implication → основной текст и key_takeaway — из resolution → trigger → top_weight
narrative_role   → приоритет при выборе anchor-сигнала
```

**Правило управления tension кластера:**
Блок «Главные нарративы» показывает tension победителя — сигнала с максимальным `links.contradicts`. Чтобы сменить tension кластера:
1. Добавить **новый** сигнал с сильным `tension` по правилам Шага 6 — не редактировать `tension` уже существующего сигнала (Immutability Policy, `docs/NIES.md` AD-7: `tension`/`macro_implication` immutable после первого синтеза, в котором сигнал использован)
2. Обеспечить у нового сигнала больше `links.contradicts` чем у текущего победителя — **только через честный тест Шага 5**, накрутка запрещена

## База артефактов (ENTITIES.json)

При разборе каждого сигнала Claude выявляет значимые сущности и обновляет `ENTITIES.json`.

### Типы артефактов

| type | Примеры |
|------|---------|
| `l2` | Stacks, Rootstock, Citrea, Botanix |
| `protocol` | Lightning, Stratum V2, wBTC |
| `corporate` | Strategy, GoMining, Bitplanet, Nakamoto |
| `fund` | Polychain Capital, BlackRock, Fidelity |
| `infrastructure` | Antalpha, Bitmain, Chainlink, DMND |
| `exchange` | Coinbase, OKX, Arkham |

### Схема объекта

```json
{
  "id": "stacks",
  "name": "Stacks",
  "type": "l2",
  "status": "active",
  "summary": "Одна строка — суть сущности",
  "profile": {
    "what": "Что делает",
    "metrics": ["ключевая метрика", "..."],
    "links_to": ["bitcoin", "ethereum"],

  "signal_refs": ["INF-2026-0609-001"],
  "last_updated": "YYYY-MM-DD"
}
```

### Правила обновления

- **Новая сущность** → создать объект и добавить в `ENTITIES.json`
- **Известная сущность** → обновить `profile` свежими данными + добавить `signal_ref` + обновить `last_updated`
- **Критерий значимости** → сущность попадает в базу если упоминается в контексте сигнала как участник события, а не просто как сравнение
- `status`: `active` / `closed` / `pending`

## HOLDERS_DATA — правило верификации (data/holders.js)

`data/holders.js` — снэпшоты распределения BTC по 6 категориям (`individuals`, `lost_satoshi`, `governments`, `companies`, `etfs`, `not_mined`) на конкретные даты. Используется таблицей «Структура владения» на сайте.

**Правило:** если в ходе обработки сигнала (Шаг 2 — Исследование) встречается свежая цифра по сумме владения любой из этих категорий — она проверяется и, если данные обновились, добавляется новый снэпшот (или уточняется существующий, если снэпшот той же даты уже есть). Это не про сам сигнал — сигнал обрабатывается по обычному алгоритму Шагов 1–8; снэпшот HOLDERS_DATA — побочный, но обязательный апдейт, если попавшиеся цифры релевантны.

### Методология категорий

- **`individuals` — residual**, не независимо измеренная величина: `21 000 000 − (lost_satoshi + governments + companies + etfs + not_mined)`. Так устроены все предыдущие снэпшоты (сумма категорий равна 21,000,000 по построению) — новый снэпшот обязан сохранять эту же логику, иначе датасет станет внутренне противоречивым.
- **`not_mined`** — считается напрямую из текущего блока эмиссии, не требует стороннего источника для основного значения; для перепроверки годятся live-трекеры circulating supply (Newhedge, MacroMicro, Blockchain.com) — если 2+ независимых сходятся в пределах погрешности, использовать без дополнительных оговорок.
- **`lost_satoshi`, `governments`, `companies`, `etfs`** — оценочные категории, разные агрегаторы (Arkham, River Financial, BitcoinTreasuries.net, Glassnode, Chainalysis, MEXC/KuCoin-обзоры) дают несовпадающие цифры, иногда на разные даты внутри одного месяца.

### Обязательные оговорки при добавлении/обновлении снэпшота

- Если источники по одной категории расходятся — фиксировать оба значения в комментарии к записи (не выбирать молча), указывать почему выбран именно этот источник (специализация трекера важнее общего агрегатора)
- Указывать дату каждого использованного источника отдельно — снэпшот часто составной (разные категории с разных дат), это не единый пруф на одну дату
- После пересчёта `individuals` (residual) — проверять, что сумма всех категорий даёт ровно 21,000,000; если нет — арифметика неверна, искать ошибку до записи

### Приоритетные источники по категориям (для будущих обновлений)

| Категория | Предпочтительный источник |
|-----------|---------------------------|
| Companies | BitcoinTreasuries.net (уже основной источник проекта — предпочесть его общим агрегаторам типа MEXC/KuCoin, если данные доступны) |
| ETFs | BitcoinTreasuries.net, Farside Investors |
| Governments | Arkham Intelligence (специализированный трекер, предпочесть общим обзорам) |
| Individuals/whales | BitInfoCharts rich list, Glassnode |
| Lost/Satoshi | Chainalysis (оригинальная методология), Glassnode |

## FACTS — машиночитаемые факты (целевая архитектура, v8.2)

**Проблема, которую решает этот раздел:** BDKS и NIES уже декларируют принцип Single Source of Truth — «корпус сигналов — единственная база записей; производные базы ссылаются, не копируют» (NIES, S12: «направление синхронизации неизменно — производное приводится к источнику»). На практике этот принцип массово нарушался: `index.html` и `ENTITIES.json.profile.metrics` годами копировали числа текстом вместо ссылки на сигнал-источник. Конкретный подтверждённый случай: `signals.json.meta.strategy_btc_reserve` (847 363, обновлено вручную 25 июня) разошёлся с реальным фактом из сигнала STR-2026-0706-001 (843 775, 5 июля) — расхождение никто не поймал бы, если бы его не заметили руками.

**Решение:** опциональное поле `facts` в схеме сигнала (см. «Сигнал: схема объекта» выше) — структурированное дополнение к свободнотекстовому `data`, не замена. `scripts/build_facts.py` сканирует все сигналы, для каждого `key` выбирает запись с максимальным `as_of` (не порядок в файле, не дата сигнала) и пишет `data/facts.json`. Этот файл — производный, регенерируется автоматически (`.github/workflows/build-facts.yml` при каждом изменении `signals.json`), **никогда не редактируется руками** — тот же принцип, что уже действует для `SIGNALS.md`.

### Когда заполнять `facts[]`

Если сигнал содержит числовой факт по сущности или категории, которая уже отображается (или в перспективе может отображаться) где-то на сайте — заполнить `facts[]` тем же коммитом, что и сам сигнал. Не обязательно для каждого числа в `data` — только для тех, что имеют смысл как отслеживаемая метрика с историей обновлений (холдинги сущности, доля категории владения и т.п.), не для одноразовых контекстных цифр.

### Формат ключа

`entity_id.metric` — `entity_id` переиспользует `id` из `ENTITIES.json` там, где сущность там существует (например `strategy`); для агрегатных категорий без отдельной сущности (`top100_public_companies`, `governments`, `etf_exchanges` и т.п.) — любой стабильный snake_case идентификатор. Оба сегмента — snake_case, схема (`schemas/signal/v1.json`) валидирует формат через regex.

### Что НЕ проходит через этот механизм

- **Детерминированные/протокольные числа** (эмиссия по номеру блока, халвинг-математика) — вычисляются напрямую, сигнал не нужен
- **Live-fetch данные** (hashrate, цена, объём торгов) — уже есть рабочий пайплайн (mempool.space, CoinGecko via `data/volume.json`), не нужно тащить через сигналы

### Использование на сайте

`index.html` вызывает `fetchFacts()` при инициализации → `window.FACTS[key]` → `{value, unit, as_of, signal_id}`. Приоритет разрешения значения там, где раньше было захардкожено или бралось из `meta`: **facts.json → signals.json.meta (устаревающий вручную) → константа-fallback**.

**Универсальный механизм для HTML:** любой элемент с `data-fact-key="entity_id.metric"` автоматически подхватывает значение через `applyFactsToDOM()` при каждом `fetchFacts()` — не нужно писать код на каждое новое место миграции, достаточно проставить атрибут. Оборачивать только само число в разметке, текст вокруг не трогать (`data-fact-format="k"` — сокращённый вид, например «844K»).

**Синхронизация TREASURY_HOLDERS.json:** `scripts/build_facts.py` также сверяет поле `btc` у 100 публичных компаний-держателей с фактами по явному маппингу имён (`TREASURY_NAME_TO_FACT_KEY` — намеренно explicit, не fuzzy-match, чтобы не смэтчить не ту компанию). Расхождение логируется (`::warning::`) и правится — обнаружено и исправлено сразу 3 случая (Strategy, Strive, OranjeBTC) при первом прогоне на реальных данных.

### Защита от расхождения (урок AD-6 — тест-страж сразу, не постфактум)

`tests/unit/test_facts.py` — проверяет: (1) внутри одного сигнала нет дублирующихся `facts[].key`; (2) `resolve_facts()` корректно выбирает запись с максимальным `as_of` независимо от порядка сигналов в файле; (3) `sync_treasury_holders()` правит только замапленные записи и не трогает остальные. `tests/unit/test_contract_schemas.py` автоматически валидирует `facts[]` каждого сигнала против JSON Schema.

**Фаза 5 — сторожевой аудит против повторного дрейфа.** Миграция (Фазы 1–4) была процедурной — числа искались вручную. Процедура без механизма проверки в этом проекте исторически не держится (AD-6). `scripts/check_stale_facts.py` берёт **полную историю** значений каждого отслеживаемого ключа (`build_facts.superseded_values()` — не только текущее по `as_of`, а все когда-либо встреченные) и ищет любое из устаревших значений голым текстом в `index.html`/`TREASURY_HOLDERS.json` — вне `data-fact-key` (там placeholder безопасен, перезаписывается JS) и вне JS-констант fallback (`const NAME = число;` — намеренно ниже `facts.json` в приоритете, не дублирующее отображение). Оформлен как pytest-тест (`test_no_forgotten_stale_fact_copies_in_index_html`), не отдельный CI-шаг — попадает в основной прогон автоматически, не требует отдельного подключения в workflow, которое можно забыть. Проверен на подставленном кейсе (`test_check_stale_facts_catches_injected_stale_copy`) — детектор действительно ловит, не просто молча проходит на чистом репозитории.

При первом реальном прогоне сразу нашёл забытое место (`mstr-live-btc` — обновлялось отдельным JS-механизмом напрямую по `id`, не через `data-fact-key`) — унифицировано на единую конвенцию вместо добавления исключения в сторож.

### Текущий охват

10 ключей, 9 сущностей: `strategy` (btc_holdings, usd_reserve), `metaplanet`, `twenty_one_capital`, `mara_holdings`, `bstr`, `strive`, `oranjebtc`, `el_salvador` — все с `btc_holdings`, плюс 2 агрегатные категории (`top100_public_companies`, `etf_exchanges`). Найдено и устранено 11 реальных расхождений в проде на момент миграции: 8 в `index.html` (Strategy ×7, Metaplanet ×1), 3 в `TREASURY_HOLDERS.json` (Strategy, Strive, OranjeBTC). Расширение на оставшиеся ~15 сущностей ENTITIES.json без явного числового дублирования на сайте — низкий приоритет, по мере появления новых мест отображения.

## SITE_MAP — карта структуры сайта (v8.5)

**Проблема:** за сессию несколько раз находки о дублировании/устаревании контента обнаруживались случайно, не системно (график эмиссии дублировал 3 места, панель уезжала не в тот кластер, футер не обновлялся из-за неправильного триггера вкладки) — потому что не было машиночитаемой карты структуры сайта, только сам `index.html` на 7000+ строк. Та же природа проблемы, что решал FACTS для данных — здесь решается для структуры.

**`data/site_map.json`** — единственный источник истины по структуре: на каждую панель/раздел — кластер, вкладка, тип контента (`kind`), происхождение данных (`source`), ключевые слова для поиска. Заполняется вручную (авто-извлечение из разнородной разметки ненадёжно), но **сверяется тестом в обе стороны**:

- `test_every_manifest_title_exists_in_html` — ловит переименованную/удалённую панель, если манифест не обновили
- `test_every_html_panel_title_is_in_manifest` — ловит новую панель, которую забыли занести в манифест

Записи с `"dynamic": true` (например «Дайджест сигналов по кластерам» — целиком JS-рендерится, нет статичного literal-текста) исключены из первой проверки — по построению нечего сверять.

### Значения `source`

| Значение | Смысл |
|----------|-------|
| `live` | живой fetch при каждой загрузке (mempool.space, Yahoo Finance) |
| `facts` | из механизма FACTS (`data/facts.json`) или синхронизируемых JSON (`TREASURY_HOLDERS.json`, `data/holders.js`) |
| `static` | статичные данные, обновляются только вручную |
| `signals` | нарративный синтез из корпуса сигналов, не число |
| `none` | методология/описание, живых данных нет |
| `tool` | интерактивный инструмент (AI-анализатор), не витрина данных |

### UI — интерактивное дерево

Кнопка 🔍 в шапке → полноэкранный оверлей: поиск по названию/ключевым словам/вкладке, дерево кластер→вкладка→запись с цветным бейджем `source`, клик — переход на вкладку + скролл к панели. Переход ищет панель **по точному тексту заголовка** (`.panel-title`/`.acc-label`/`.instrument-ticker`), не по `id` — расставлять уникальные id на полсотни существующих панелей ради одной этой функции было бы избыточным риском для минимальной пользы; текстовый поиск работает надёжно и не требует трогать разметку панелей вообще.

### При добавлении новой панели на сайт

Добавить запись в `data/site_map.json` (title — точное совпадение с `panel-title`/`acc-label` в HTML, cluster/tab — куда встроена, source — по таблице выше, keywords — по смыслу) тем же коммитом. Тест поймает несоответствие, если забыть — но лучше не полагаться только на это.
