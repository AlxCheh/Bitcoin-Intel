## Добавление сигнала

Каждый новый сигнал — **два файла одновременно**: `signals.json` + `SIGNALS.md`.

### Полная схема объекта сигнала

```json
{
  "id": "STR-2026-0623-001",
  "date": "YYYY-MM-DD",
  "cat": "ownership",
  "catLabel": "🏛️ ВЛАДЕНИЕ",
  "dir": "neg",
  "horizon": "mid",
  "theme": "institutionalization",
  "weight": "media",
  "actor": "corporate",
  "flow": "outflow",
  "signal": "Заголовок сигнала",
  "data": ["метрика 1", "метрика 2"],
  "context": "Исторический контекст и аналогии.",
  "caveat": "Риски и ограничения интерпретации.",
  "source": "Источник (месяц год)",
  "links": {
    "confirms": ["id или заголовок сигнала который этот усиливает"],
    "contradicts": ["id или заголовок сигнала с которым в противоречии"],
    "context_chain": ["id сигнала из контекста которого вырастает этот"]
  },
  "narrative_role": "complication",
  "cluster": "strategy_model_stress",
  "tension": "краткое описание противоречия которое сигнал раскрывает",
  "macro_implication": "что означает на уровне общей картины"
}
```

> `index.html` для сигналов не трогать — данные загружаются из `signals.json` через fetch.

## Метаметки сигнала

**`id`** — уникальный идентификатор по формату `PREFIX-YYYY-MMDD-NNN`:

| Префикс | Тема |
|---------|------|
| `STR` | institutionalization |
| `SUP` | supply |
| `INF` | infrastructure |
| `MAC` | macro |
| `NAR` | narrative |

**`dir`** — что сигнал говорит о силе Bitcoin как актива в своём горизонте:

| Значение | Смысл |
|----------|-------|
| `pos` | усиливает позицию BTC |
| `neg` | создаёт давление на BTC |
| `neu` | структурный факт без направления |

**`horizon`** — временной горизонт:

| Значение | Смысл |
|----------|-------|
| `short` | недели–месяцы |
| `mid` | месяцы–год |
| `long` | годы, структурные изменения |

**`theme`** — смысловой кластер:

| Значение | Смысл |
|----------|-------|
| `supply` | дефицит, эмиссия, потерянные монеты |
| `institutionalization` | ETF, корпорации, правительства, владение |
| `infrastructure` | Lightning, Layer 2, майнинг |
| `macro` | ставки, ликвидность, макро-давление |
| `narrative` | философия, регуляция, нарративы |

**`weight`** — достоверность источника:

| Значение | Смысл |
|----------|-------|
| `onchain` | верифицируемые данные блокчейна |
| `market` | биржевые данные, цена, объёмы |
| `media` | медиа, аналитики, оценки |
| `primary` | первичный источник (высказывание, документ) |

**`actor`** — субъект сигнала:

| Значение | Смысл |
|----------|-------|
| `etf` | биржевые фонды (BlackRock, Fidelity) |
| `corporate` | публичные компании (Strategy, MSTR) |
| `government` | государственные резервы |
| `defi` | протоколы, on-chain структуры |
| `retail` | розничные инвесторы |
| `miner` | майнеры |

**`flow`** — направление капитала:

| Значение | Смысл |
|----------|-------|
| `inflow` | капитал входит в BTC |
| `outflow` | капитал уходит из BTC |
| `internal` | перераспределение внутри структуры |
| `neutral` | движения капитала нет или неизвестно |

## Связывание сигналов — правило `links`

Каждый новый сигнал обязательно анализируется на связи с уже существующими.

### Типы связей

**`confirms`** — сигнал усиливает или подтверждает другой:
> Пример: «90% от $300 млн идёт в резерв» → confirms «NAV-дисконт 0.83x» (оба говорят что модель под давлением)

**`contradicts`** — сигнал в противоречии с другим:
> Пример: «$300 млн в резерв» → contradicts «резервы 846 842 BTC превышают долг» (Сейлор говорит всё хорошо, но действия иные)

**`context_chain`** — сигнал вырастает из контекста другого:
> Пример: «90% в резерв» → context_chain «STRC создаёт структурное давление на покупку BTC» (именно STRC потребовал резерв)

### Алгоритм при получении нового сигнала

1. **Определить тему и кластер** — к какому `theme` и `cluster` относится
2. **Найти все сигналы того же `cluster`** — прочитать их `signal` и `tension`
3. **Проверить три вопроса:**
   - Какие сигналы из базы этот **усиливает**? → `confirms`
   - С какими **противоречит**? → `contradicts`
   - Из какого контекста **вырастает**? → `context_chain`
4. **Назначить `narrative_role`:**
   - `trigger` — запускает новую цепочку событий
   - `complication` — усложняет существующий нарратив
   - `resolution` — разрешает противоречие
   - `background` — структурный контекст без острого события
5. **Сформулировать `tension`** — одна фраза: что именно в противоречии
6. **Сформулировать `macro_implication`** — что это означает для BTC в целом

### Кластеры (текущие)

| cluster | Описание | Сигналов |
|---------|----------|---------|
| `strategy_model_stress` | Устойчивость модели Strategy под давлением долга | 7 |
| `etf_institutional_flow` | Потоки капитала через ETF и институциональные структуры | 4 |
| `btc_treasury_competition` | Корпоративное накопление BTC выходит за пределы Strategy | 2 |
| `btc_infrastructure_growth` | Рост инфраструктуры: Lightning, L2, транзакции | 3 |
| `supply_scarcity` | Дефицит предложения: потерянные монеты, эмиссия | 1 |

> Новый кластер создаётся когда появляется 2+ сигнала с общим `tension` которого нет в существующих.

## Схема структурирования материала

Любой входящий материал разбирать по четырём пунктам:
- **Что это** — краткое определение
- **Контекст** — исторические аналогии
- **Что означает сейчас** — интерпретация
- **Важные оговорки** — риски, ограничения

### Категории сигналов

| cat | catLabel | Примеры |
|-----|----------|---------|
| `onchain` | 📊 On-chain | Production Cost, MVRV, NVT |
| `ta` | 📈 Технический анализ | Уровни, паттерны, тренды |
| `macro` | 🏦 Макроэкономика | Ставки, ликвидность, DXY |
| `mining` | ⛏️ Майнинг | Хешрейт, сложность, капитуляция |
| `narrative` | 📰 Нарратив | Регуляция, институционалы, события |
| `layer2` | 🔗 Layer 2 | Stacks, Lightning, сайдчейны |
| `ownership` | 🏛️ Владение | Распределение по типам владельцев |

## Правило наполнения вкладок

Claude **не наполняет вкладки из собственных знаний**. Только из материалов пользователя.

1. Пользователь присылает материал
2. Claude структурирует по схеме выше
3. Переход к следующей теме — только после явного сигнала пользователя
