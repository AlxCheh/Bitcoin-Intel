## Правило: изменения index.html

Любой запрос на изменение **структуры или визуала сайта** (`index.html`) — новые блоки, вкладки, дизайн, компоненты — требует предварительного чтения алгоритма сборки:

→ [docs/spec-pilot.md](docs/spec-pilot.md)

**Когда читать:** запрос содержит «добавь», «сделай», «перенеси», «измени» применительно к `index.html` или к тому как сайт выглядит или работает.

**Когда не читать:** правка сигналов (`signals.json`, `SIGNALS.md`), обновление документации, мелкие однострочные правки с очевидным решением.

## Сигнал: схема объекта

Каждый новый сигнал — **два файла одновременно**: `signals.json` + `SIGNALS.md`.

```json
{
  "id": "STR-2026-0623-001",
  "date": "YYYY-MM-DD",
  "cat": "ownership",
  "catLabel": "🏛️ ВЛАДЕНИЕ",
  "dir": "neg",
  "horizon": "mid",
  "theme": "institutionalization",
  "weight": "media",
  "actor": "corporate",
  "flow": "outflow",
  "signal": "Заголовок сигнала",
  "data": ["метрика 1", "метрика 2"],
  "context": "Исторический контекст и аналогии.",
  "caveat": "Риски и ограничения интерпретации.",
  "source": "Источник (месяц год)",
  "links": {
    "confirms": ["id сигнала который этот усиливает"],
    "contradicts": ["id сигнала с которым в противоречии"],
    "context_chain": ["id сигнала из контекста которого вырастает этот"]
  },
  "narrative_role": "complication",
  "cluster": "strategy_model_stress",
  "tension": "краткое описание противоречия которое сигнал раскрывает",
  "macro_implication": "что означает на уровне общей картины"
}
```

- Все три поля `links` — массивы **id** (не заголовков)
- `index.html` для сигналов не трогать — данные загружаются из `signals.json` через fetch

## Справочник значений полей

> Формат и допустимые значения — здесь. Доменный смысл значений (связь с онтологией) → [docs/DATA_DICTIONARY.md](docs/DATA_DICTIONARY.md).

**`id`** — формат `PREFIX-YYYY-MMDD-NNN`. Префикс выбирается по `theme` **на момент создания**:

| Префикс | theme | Смысл темы |
|---------|-------|-----------|
| `SUP` | `supply` | Дефицит, эмиссия, потерянные монеты |
| `STR` | `institutionalization` | ETF, корпорации, правительства, владение |
| `INF` | `infrastructure` | Lightning, Layer 2, майнинг |
| `MAC` | `macro` | Ставки, ликвидность, макро-давление |
| `NAR` | `narrative` | Философия, регуляция, нарративы |

**Правило неизменности `id`:** после записи `id` не меняется никогда — на него ссылаются `links` других сигналов. Если `theme` сигнала уточнилась позже, префикс остаётся историческим; расхождение префикса и темы задним числом не исправляется.

**`dir`** — что сигнал говорит о силе Bitcoin как актива в своём горизонте:

| Значение | Смысл |
|----------|-------|
| `pos` | усиливает позицию BTC |
| `neg` | создаёт давление на BTC |
| `neu` | структурный факт без направления |

**`horizon`** — временной горизонт:

| Значение | Смысл |
|----------|-------|
| `short` | недели–месяцы |
| `mid` | месяцы–год |
| `long` | годы, структурные изменения |

**`weight`** — достоверность источника:

| Значение | Смысл |
|----------|-------|
| `onchain` | верифицируемые данные блокчейна |
| `market` | биржевые данные, цена, объёмы |
| `media` | медиа, аналитики, оценки |
| `primary` | первичный источник (высказывание, документ) |

**`actor`** — субъект сигнала:

| Значение | Смысл |
|----------|-------|
| `etf` | биржевые фонды (BlackRock, Fidelity) |
| `corporate` | публичные компании (Strategy, MSTR) |
| `government` | государственные резервы |
| `defi` | протоколы, on-chain структуры |
| `retail` | розничные инвесторы |
| `miner` | майнеры |

**`flow`** — направление капитала:

| Значение | Смысл |
|----------|-------|
| `inflow` | капитал входит в BTC |
| `outflow` | капитал уходит из BTC |
| `internal` | перераспределение внутри структуры |
| `neutral` | движения капитала нет или неизвестно |

**`cat` / `catLabel`** — категория и её отображаемая метка. `catLabel` выводится из `cat` строго по таблице — никаких вариаций регистра, эмодзи или синонимов:

| cat | catLabel (канон) | Примеры |
|-----|------------------|---------|
| `onchain` | 📊 ON-CHAIN | Production Cost, MVRV, NVT |
| `macro` | 🏦 МАКРО | Ставки, ликвидность, DXY |
| `mining` | ⛏️ МАЙНИНГ | Хешрейт, сложность, капитуляция |
| `narrative` | 📰 НАРРАТИВ | Регуляция, институционалы, события |
| `layer2` | 🔗 LAYER 2 | Stacks, Lightning, сайдчейны |
| `ownership` | 🏛️ ВЛАДЕНИЕ | Распределение по типам владельцев |

> Категория `ta` (технический анализ) депрецирована в v6.0: технический анализ выведен за границы предметной области (обоснование → [docs/BDKS.md](docs/BDKS.md), Раздел 1). Исторических сигналов с этой категорией нет; новые не создаются.

## Алгоритм обработки сигнала

**Правило видимости рассуждения:**
Шаги 3 (Разбор), 4 (Кластеризация), 5 (Связывание) и 6 (Синтез) содержат интерпретационные решения, не механические проверки: выбор `narrative_role`, формулировка `tension`/`macro_implication`, состав `confirms`/`contradicts`/`context_chain`, причина выбора `cluster`. Эти решения и их обоснование показываются пользователю в чате до сборки итогового JSON — отдельным разбором по шагам, а не только результатом.

«Сводка после записи сигнала» (см. ниже) — краткое резюме уже показанных и принятых решений, а не единственное место, где они становятся видны. Свернуть весь процесс сразу в эту сводку недопустимо, даже если шаблон сводки кажется самодостаточным.

```
✗ Claude молча проходит Шаги 3–6 внутри себя, показывает только готовый JSON + сводку
✓ Claude показывает разбор (что это / контекст / что значит / оговорки), затем
   обоснование cluster и links, и только потом — оформление и сводку
```

### Шаг 1 — Приём материала

Определить формат входящего материала: текст, ссылка, скриншот.
Зафиксировать что именно прислал пользователь — это исходная точка, не интерпретация.

**Ограничение:** Claude не наполняет сигналы из собственных знаний. Только из материала пользователя. Переход к следующему сигналу — только после явного подтверждения пользователя.

### Шаг 2 — Исследование

Найти первоисточник:
- Если дана ссылка → `fetch` по ней
- Если нет → `web_search` по ключевым фактам материала

Найти 2–3 дополнительных источника по типу сигнала:

| Тип сигнала | Источники |
|------------|-----------|
| Корпоративные покупки BTC | Arkham, SEC filing, пресс-релиз |
| ETF / регуляторика | SEC.gov, Bloomberg ETF, проспект |
| Майнинг / инфраструктура | Hashrate Index, Glassnode, отчёты |
| Макро / on-chain | Glassnode, CryptoQuant, VanEck ChainCheck |
| Нарративы / медиа | Bitcoin Magazine, CoinDesk, The Block |

- Если источники противоречат — зафиксировать расхождение, оно войдёт в `caveat`
- Если первоисточник недоступен — сообщить пользователю и работать с тем что есть

### Шаг 3 — Разбор

Структурировать собранный материал по четырём пунктам:

- **Что это** — краткое определение события или факта
- **Контекст** — исторические аналогии, предыдущие прецеденты
- **Что означает сейчас** — интерпретация для BTC
- **Важные оговорки** — риски, ограничения, противоречия источников

Это читаемый промежуточный артефакт. Из него питаются поля JSON:

| Пункт разбора | Поля JSON |
|--------------|-----------|
| Что это | `signal`, `cat`, `catLabel`, `actor`, `flow` |
| Контекст | `context` |
| Что означает сейчас | `dir`, `horizon`, `weight`, `theme` |
| Важные оговорки | `caveat`, `data` |

Поля `id`, `date`, `source` заполняются отдельно: `source` — название источника и месяц, `id` — на шаге 8.

**Правило определения `date`:**
`date` — дата события, берётся из самого проверенного источника, найденного на Шаге 2, а не выводится расчётом от относительных формулировок пользователя («вчера», «сегодня», «на этой неделе») и не от даты сессии.

Приоритет источника даты:
1. Дата/период в самом первоисточнике (официальный релиз, regulatory filing, документ) — высший приоритет
2. Дата публикации наиболее авторитетного вторичного источника (Шаг 2, колонка «Источники»)
3. Если ни один источник дату не называет явно — спросить пользователя напрямую, не предполагать

```
✗ Пользователь: «отчиталась вчера» → Claude молча вычисляет: сегодня минус 1 день
✓ Claude берёт дату из самого filing/источника; если расходится с «вчера» пользователя —
   фиксирует это расхождение и уточняет, а не выбирает дату самостоятельно
```

Если источники расходятся в дате — расхождение идёт в `caveat`, используется дата самого первичного источника.

### Шаг 4 — Кластеризация

Определить `theme` по справочнику полей.

Определить `cluster` по двум признакам одновременно:
1. `theme` совпадает с темой кластера
2. Субъект события (`actor` + суть действия) совпадает с описанием кластера

Если оба признака совпадают → кластер найден.
Если только `theme` совпадает → потенциально новый кластер (создаётся при 2+ сигналах с общим противоречием которого нет в существующих).

Кластер назначается здесь. `tension` формулируется только на шаге 6 — не использовать его как критерий поиска на этом шаге.

### Шаг 5 — Связывание

Найти кандидатов: все сигналы с тем же `theme`. Если кластер определён — добавить все сигналы того же кластера.

Для каждого кандидата проверить три критерия:

**`confirms`** — оба сигнала говорят одно и то же но через разные механизмы?
→ Если да: добавить в `confirms`

**`contradicts`** — `macro_implication` двух сигналов описывают **несовместимые состояния одной системы**?
Тест: можно ли одновременно быть правым и в A, и в B?
→ Нет → добавить в `contradicts`
→ Да → не добавлять

**Запрещено** добавлять `contradicts` чтобы сделать сигнал победителем tension кластера. Чтобы сменить tension кластера — переписать поле `tension` у нужного сигнала (см. «Нарративный синтез»).

**`context_chain`** — без понимания сигнала X текущий сигнал теряет смысл?
→ Если да: добавить в `context_chain`

Пустое поле — норма если после проверки всех кандидатов критерий не выполнен ни разу.
Не добавлять связь только потому что сигналы близки по теме.

### Шаг 6 — Синтез

Выполняется только после завершения шага 5.

**`tension`** — противоречие между двумя силами, не описание факта:
- Формула: «X vs Y» / «X несмотря на Y» / «X при условии что Y»
- Два конкретных механизма в столкновении
- Всегда начинается с заглавной буквы

| ❌ Плохо | ✅ Хорошо |
|---------|---------|
| «Strategy продолжает покупать BTC» (нет противоречия) | «Strategy наращивает долг для покупки BTC vs рынок ставит NAV-дисконт 0.83x» |
| «ETF показал отток» | «ETF-оттоки как поверхностное давление vs долгосрочные держатели покупают в 10x больше» |

**`macro_implication`** — ответить на вопрос «что это меняет для BTC структурно».
Не пересказ события — вывод об изменении в системе.

| ❌ Плохо | ✅ Хорошо |
|---------|---------|
| «Franklin подала заявку на ETF» | «Пассивный дивидендный поток как источник BTC-спроса — новая категория не зависящая от настроений рынка» |
| «Metaplanet купила Siiibo» | «BTC-казначейство эволюционирует от пассивного баланса к операционному движку: коллатерал → лицензия → продукты → комиссионный доход» |

**`narrative_role`** — выбрать одну роль:

| Роль | Критерий | Расшифровка для сводки |
|------|---------|------------------------|
| `trigger` | Первый сигнал нового кластера или резкий перелом | открывает цепочку |
| `complication` | Добавляет противоречие в существующий нарратив | усложняет нарратив |
| `resolution` | Закрывает `tension` предыдущих сигналов | закрывает противоречие |
| `background` | Структурный контекст без острого события | структурный контекст |

### Шаг 7 — Проверка

Сигнал готов к оформлению только если:

- `tension` сформулирован по формуле (не описание факта)
- `macro_implication` отвечает на «что меняется структурно» (не пересказ)
- `narrative_role` выбрана и обоснована
- `links` — все три поля проверены по критериям шага 5

Если что-то не выполнено — вернуться к соответствующему шагу.

### Шаг 8 — Оформление

**Присвоить `id`:**
Формат `PREFIX-YYYY-MMDD-NNN`. Перед записью открыть `signals.json` и найти все существующие `id` с той же датой и тем же префиксом. `NNN` — следующий порядковый номер. Если `STR-2026-0626-001` уже есть → `STR-2026-0626-002`.

**Чеклист перед записью:**
- JSON валиден — нет незакрытых скобок, нет лишних запятых
- `id` уникален в `signals.json`
- `catLabel` соответствует `cat` по канонической таблице справочника
- Все обязательные поля заполнены (прошёл шаг 7)
- `ENTITIES.json` обновлён, если в сигнале участвуют значимые сущности (см. «База артефактов»)
- `signals.json` и `SIGNALS.md` обновляются в одном коммите
- Коммит использует префикс `signal:` (не `feat:`/`fix:`), формат —
  см. [`CONTRIBUTING.md`](CONTRIBUTING.md#commit-message-convention)

`index.html` не трогать.

## Сводка после записи сигнала

После записи каждого сигнала в `signals.json` и `SIGNALS.md` Claude выдаёт сводку в формате:

```
✓ {id} записан

Сигнал {narrative_role} в кластере {cluster}
«{описание кластера из таблицы}».
{1–2 предложения: что произошло и почему это важно структурно}

Противоречит: {id} ({краткий смысл — почему противоречие})

Подтверждает: {id} ({краткий смысл}) и {id} ({краткий смысл}) —
{1 предложение: что складывается в единую картину}

Вырастает из: {id} ({краткий смысл}) —
{1 предложение: почему этот контекст важен}

cat: {catLabel} · dir: {dir} · horizon: {horizon}
Кластер: {cluster} · {N} сигналов
Роль: {narrative_role} — {одна фраза что это означает}
```

**Правила формата:**
- Пустые `links` не выводить (если `confirms` пустой — строку не показывать)
- Описание кластера — из таблицы кластеров, расшифровка роли — из таблицы Шага 6
- `{N}` — фактическое число сигналов кластера в `signals.json` после записи

## Нарративный синтез (кратко)

Полная документация → [ALGORITHM.md](docs/ALGORITHM.md). Карточка нарратива строится из полей сигналов — алгоритм не генерирует текст, он **выбирает лучшее** из написанного.

**Score сигнала** (формула `_score_signal`, ALGORITHM.md):

| Компонент | Значения |
|-----------|----------|
| Свежесть | ≤7 дней = **+3** · ≤30 дней = **+1** · старше = 0 |
| `weight` | `onchain` = **+4** · `primary` = **+3** · `market` = **+2** · `media` = **+1** |
| `narrative_role` | `trigger` = **+4** · `complication` = **+3** · `resolution` = **+2** · `background` = 0 |
| `links.contradicts` | **+5 за каждый id** в списке (без верхнего предела) |

> `tension` в score не участвует — он определяет текст карточки, не ранжирование.

**Что определяет текст карточки:**

```
tension          → главный крючок (золотая полоса) — из сигнала с max contradicts
macro_implication → основной текст и key_takeaway — из resolution → trigger → top_weight
narrative_role   → приоритет при выборе anchor-сигнала
```

**Правило управления tension кластера:**
Блок «Главные нарративы» показывает tension победителя — сигнала с максимальным `links.contradicts`. Чтобы сменить tension кластера:
1. Написать сильный `tension` по правилам Шага 6
2. Добавить в `links.contradicts` больше id чем у текущего победителя — **только через честный тест Шага 5**, накрутка запрещена

## База артефактов (ENTITIES.json)

При разборе каждого сигнала Claude выявляет значимые сущности и обновляет `ENTITIES.json`.

### Типы артефактов

| type | Примеры |
|------|---------|
| `l2` | Stacks, Rootstock, Citrea, Botanix |
| `protocol` | Lightning, Stratum V2, wBTC |
| `corporate` | Strategy, GoMining, Bitplanet, Nakamoto |
| `fund` | Polychain Capital, BlackRock, Fidelity |
| `infrastructure` | Antalpha, Bitmain, Chainlink, DMND |
| `exchange` | Coinbase, OKX, Arkham |

### Схема объекта

```json
{
  "id": "stacks",
  "name": "Stacks",
  "type": "l2",
  "status": "active",
  "summary": "Одна строка — суть сущности",
  "profile": {
    "what": "Что делает",
    "metrics": ["ключевая метрика", "..."],
    "links_to": ["bitcoin", "ethereum"],
    "notable": "Ключевой факт или отличие"
  },
  "signal_refs": ["INF-2026-0609-001"],
  "last_updated": "YYYY-MM-DD"
}
```

### Правила обновления

- **Новая сущность** → создать объект и добавить в `ENTITIES.json`
- **Известная сущность** → обновить `profile` свежими данными + добавить `signal_ref` + обновить `last_updated`
- **Критерий значимости** → сущность попадает в базу если упоминается в контексте сигнала как участник события, а не просто как сравнение
- `status`: `active` / `closed` / `pending`
