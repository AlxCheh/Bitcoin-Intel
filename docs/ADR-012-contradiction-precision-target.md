# ADR-012: Целевое значение Contradiction Precision снижено с 85% до 60%

**Статус:** Принято
**Дата:** 2026-07-01
**Авторы:** Команда Bitcoin Intel
**Контекст ревью:** IRR v1.0 / C2 (docs/IRR_REPORT_v1.md), IRP v1.0 (docs/IRP_v1.md, REM-C2)

## Контекст

`docs/BLUEPRINT.md` §10 задаёт целевую метрику:

> Contradiction Precision (доля реальных противоречий среди добавленных) > 85%

Это значение было зафиксировано на этапе проектирования, до появления
Golden Dataset для `contradiction_detector.py`. Фактический регрессионный тест
(`tests/unit/test_contradiction.py::test_precision_on_golden_pairs`) проверяет
**>= 60%** против `tests/golden/fixtures/contradiction_pairs.json` — 73 пары
(35 positive, 38 negative), при `CONTRADICTION_PROPOSAL_THRESHOLD = 0.5`.

IRR v1.0 (C2) зафиксировал: разрыв в 25 п.п. между задекларированной целью в
BLUEPRINT и реально проверяемым порогом в тестах нигде не был явно принят как
инженерное решение. ARR v3 отметил разрыв как риск, но ARR Execution Status
пометил критерий как PASS — то есть статус проекта противоречил собственной
документации.

## Решение

Целевое значение Contradiction Precision для текущей стадии проекта — **60%**,
не 85%. 85% остаётся зафиксированным в BLUEPRINT §10 как aspirational target
для зрелой стадии (см. «Дальнейшая работа» ниже), но не является текущим DoD-критерием.

`config/settings.py::CONTRADICTION_PROPOSAL_THRESHOLD = 0.5` не меняется —
порог классификации (score >= 0.5 → «предложить как contradiction») и целевая
precision (доля верных среди предложенных) — разные величины, путать их нельзя.

## Обоснование

Тот же паттерн, что и в ADR-011 (калибровка confidence отложена):
статистически осмысленная оценка precision требует датасета такого размера,
чтобы разница в несколько ошибок не сдвигала итоговый процент на десятки
пунктов. 85% на 73 парах означает допустимо не более ~11 ошибок; 60% —
не более ~29. При текущем размере Golden Dataset требование 85% было бы не
более обоснованным, чем 60% — оба числа чувствительны к составу выборки,
но 60% оставляет запас, не создающий ложного ощущения точности.

Зафиксировать 85% как проверяемый критерий сейчас означало бы либо:
(a) искусственно расширять fixture пока не наберётся нужный процент —
подгонка данных под целевое число, либо
(b) держать тест красным как норму — что обесценивает сигнал CI.
Оба варианта хуже явного снижения порога с обоснованием.

## Дальнейшая работа

Порог **60% — не постоянный**, а gate для пересмотра:

- При достижении **N >= 150** пар в Golden Dataset (иначе выборка
  недостаточна для повышения планки без риска переобучения на шум) —
  пересчитать фактическую precision на новой выборке.
- Если фактическая precision на расширенной выборке >= 85% —
  поднять порог теста до 85%, зафиксировать новым ADR со ссылкой на этот.
- Если ниже — рассмотреть доработку `semantic_inverse_score`
  (`scripts/contradiction_detector.py`) прежде чем поднимать порог теста.
