# ADR-013: Synthesis Schema v1 отражает реальную реализацию, не ADDENDUM §17.3 дословно

**Статус:** Принято
**Дата:** 2026-07-01
**Авторы:** Команда Bitcoin Intel
**Контекст ревью:** IRP v1.0 Wave 2 / REM-B3 (docs/IRP_v1.md)

## Контекст

REM-B3 предписывает создать `schemas/synthesis/v1.json` как копию спецификации
из `docs/BLUEPRINT_ADDENDUM.md` §17.3. При попытке валидации реальных записей
`synthesis_store/*.json` (25 файлов) против этой спецификации обнаружено:
100% записей не проходят валидацию, потому что все required-поля из §17.3 —
`version`, `algorithm_version`, `ontology_version`, `window_days`,
`computed_at` — отсутствуют в каждой без исключения реальной записи.
Вместо `computed_at` код (`scripts/synthesizer.py`) пишет `generated_at`.

Это не единичный пропуск — это систематический разрыв между тем, что §17.3
спроектировал (схема версионирования алгоритма, окна расчёта, timestamp
вычисления), и тем, что реализовано. Проектирование версионирования
алгоритма — отдельная, более крупная задача, уже выделенная в IRP как
**M07 (Wave 3): `ALGORITHM_VERSION` не реализован как semver-константа** и
**M01 (Wave 3): переименование `synthesis_cache_builder.py`** — обе связаны
с той же незавершённой частью архитектуры.

## Решение

`schemas/synthesis/v1.json` описывает **текущую реализацию** (поля, которые
реально пишет `synthesizer.py` во все 25/25 существующих записей), а не
целевую спецификацию §17.3. Расхождения зафиксированы явно в самой схеме
(поле `description` на верхнем уровне и на каждом отклонившемся свойстве).

§17.3 в ADDENDUM **не изменяется** — она остаётся целевой спецификацией для
Wave 3, когда `algorithm_version`/`ontology_version`/`window_days`/`version`
будут реализованы (вместе с M07). После этого `schemas/synthesis/v1.json`
обновляется вместе с кодом, и это расхождение закрывается.

## Обоснование

Тот же паттерн, что в ADR-011 и ADR-012: контракт, который на 100% не
проходит валидацию против реальных данных, не защищает ничего — CI будет
красным с первого коммита, и следующий разработчик либо (a) обойдёт Contract
Tests целиком, либо (b) начнёт «чинить» реальные synthesis-записи под схему,
теряя реальные данные ради соответствия документу, который сам не был
реализован. Оба хуже, чем схема, которая валидирует то, что действительно
гарантирует код сегодня.

## Дальнейшая работа

При реализации M07 (Wave 3, `ALGORITHM_VERSION` как semver-константа) —
обновить `schemas/synthesis/v1.json`: добавить `algorithm_version` (и по
итогам решения M07/M06 — `ontology_version`) как required, синхронно с
кодом `synthesizer.py`. `window_days` и `version` (номер ревизии синтеза)
остаются open question — фиксировать отдельным ADR при реализации, если
будут введены.

## Связанные документы

- `docs/BLUEPRINT_ADDENDUM.md` §17.3 — целевая спецификация (не изменена)
- `docs/IRP_v1.md` REM-B3, M01, M07
- ADR-011, ADR-012 — тот же паттерн решения для аналогичных разрывов

## Обновление после M07 (2026-07-01)

При реализации REM-M07 `algorithm_version` добавлен в `schemas/synthesis/v1.json`,
но **не как required**, вопреки первоначальному плану в разделе «Дальнейшая
работа» выше. Причина: `algorithm_version` пишется только в записи,
созданные после этого коммита; 25 исторических файлов в `synthesis_store/`
поля не содержат и намеренно не бэкфилляются (это исторический архив, не
живые данные для перезаписи). Required-поле сделало бы Contract Tests
красными на всех 25 записях — тот же анти-паттерн, который эта ADR изначально
решала для `algorithm_version`/`ontology_version`/`window_days` целиком.

`ontology_version` и `window_days` из §17.3 остаются нереализованными —
не входят в это обновление, зависят от M06/открытых вопросов.
