# BDKS Knowledge Engineering Review (KER)

**Версия:** 1.0
**Объект анализа:** BDKS v1.0 (Bitcoin Domain Knowledge Specification)
**Метод:** трёхстадийный Knowledge Engineering пайплайн (Elicitation → Formalization → Validation) по практикам CommonKADS/MOKA
**Статус документа:** инженерное ревью и план рефакторинга — НЕ замена BDKS. Ни одно доменное утверждение не изобретено; все элементы формальной модели выводятся из извлечённого знания с трассировкой K-идентификаторов.

**Связанные артефакты, учитываемые при кросс-проверке:** BAMS v1.0 (docs/BAMS_v1.0.md, в main), CLAUDE.md v5.9 (уровень реализации: schema signals.json, enum'ы cat/theme/weight/actor/flow), signals.json (49 сигналов на момент ревью).

> Примечание (2026-07-02): после аудита живые документы переведены на неверсионированные имена — docs/BAMS.md, docs/BDKS.md; пути в тексте ревью отражают состояние на момент аудита.

# Часть I. Knowledge Elicitation

## I.1. Метод извлечения

Исходный BDKS v1.0 рассматривается как сырое знание (raw knowledge), а не готовая спецификация. Каждое утверждение получает идентификатор `K-NNN` с типом:

| Тип | Значение |
|---|---|
| `CON` | доменное понятие / класс |
| `ACT` | актор / участник |
| `AST` | допущение (явное или скрытое) |
| `RUL` | бизнес-правило / нормативное требование |
| `PRC` | процесс / workflow |
| `CNS` | ограничение (constraint) |
| `UNC` | зафиксированная неопределённость |
| `SRC` | источник информации |
| `PRE` | исторический прецедент |
| `MET` | метрика / показатель |
| `GOV` | правило управления знанием (governance) |

| Понятие | Где встречается | Проблема |
|---|---|---|
| **Hashrate** | Р2 (глоссарий), Р3 (класс), Р9 (метрика) | Тройное определение без кросс-ссылок; конфляция «понятие» vs «измерение понятия» |

## II.4. Формализованные карточки понятий

Шаблон (обязательные поля): Definition · Purpose · Attributes · Relationships · Constraints · Lifecycle · Dependencies · Inputs · Outputs · Validation Rules. Ниже — карточки-якоря восьми модулей (II.9); остальные 20+ классов заполняются по тому же шаблону из соответствующих строк таблицы Р3 (K-030…K-057) при рефакторинге (Часть VI).

### Card: `Protocol`
- **Definition:** набор правил консенсуса, эмиссии и валидации [K-013, K-030]
- **Purpose:** корневой инвариант домена; источник Scarcity и Verifiability [K-005, K-006]
- **Attributes:** правила консенсуса; лимит эмиссии; интервал блока
- **Relationships:** `produces` Emission; корень для всех ProtocolArtifact
- **Constraints:** не имеет владельца [K-013]; меняется только широким консенсусом [K-030]
- **Lifecycle:** квазистатичен; софт-форки с периодом «годы»
- **Validation Rules:** любое утверждение об «изменении протокола» требует ссылки на BIP-процесс [K-060]

### Card: `MiningSystem` / `Miner`
- **Definition:** механизм производства блоков (System) / экономический агент, его исполняющий (Miner) [K-033, K-081]
- **Attributes(System):** совокупная мощность → Hashrate; **Attributes(Miner):** маржа = доход − энергозатраты
- **States(Miner):** Profitable → Marginal → Capitulating → Recovering [K-033, K-081] (машина состояний II.5)
- **Relationships:** Miner `operates` MiningSystem; MiningSystem `produces` Block; Miner `holds`/`sells` BTC (давление предложения)
- **Constraints:** себестоимость энергии, Difficulty [K-081]
- **Validation Rules:** капитуляция заявляется только при подтверждении и hashrate-снижением, и on-chain продажами (двухфакторность) [K-072, K-097]

### Card: `ETFProduct` / `ETFProvider`
- **Definition:** регулируемый инструмент BTC-экспозиции / организация-эмитент [K-023, K-042, K-078]
- **Lifecycle(Product):** Filed → {Approved | Rejected} → Listed → Operating → (Delisted) [K-042] (II.5)
- **Relationships:** Provider `issues` Product; Product `custodied_by` Custodian; Product `channels` CapitalFlow
- **Inputs:** filings, дневные flows; **Outputs:** ETF Net Flows (Metric, K-106)
- **Validation Rules:** flows интерпретируются относительно AUM, не абсолютно [K-106]

### Card: `LTHCohort` / `STHCohort`
- **Definition:** когорты по возрасту неподвижности монет, порог ~155 дней — эвристика [K-021, K-049, K-050]
- **Constraints:** классификация поведенческая, не мотивационная [K-021]; наследует неопределённость кластеризации адресов [K-127]
- **Process:** цикл накопления/распределения [K-089] (II.6)
- **Validation Rules:** выводы по когортам всегда вероятностны [K-127]

### Card: `CapitalFlow`
- **Definition:** направленное движение капитала в/из BTC-экосистемы [K-047]
- **Attributes:** направление {inflow|outflow|internal}; канал {retail|institutional|derivative} [K-087]
- **Dependencies:** Liquidity (внешний макроконтур) [K-088]
- **Validation Rules:** однодневное значение не экстраполируется в тренд [K-106]

### Card: `Regulation`
- **Definition:** правовой режим по юрисдикциям [K-055]
- **Attributes:** режим {благоприятный|нейтральный|враждебный} **на юрисдикцию** — требует Jurisdiction `[GAP-FILL Q-07]`
- **Validation Rules:** регуляторное событие с долгой историей → промежуточные сигналы обязаны учитываться [K-143]

### Card: `NarrativePhenomenon` `[GAP-FILL]`
- **Definition (предлагаемая, из K-007+K-141+K-149):** устойчивый интерпретационный фрейм, влияющий на поведение агентов через рефлексивную петлю нарратив↔спрос↔освещение
- **Наблюдаемость:** через Media/Analyst сигналы (K-075) и информационные события (K-070); полная операционализация — Q-01, блокер NIES
- **Constraints:** самый низковерифицируемый уровень (Ур.4, K-146); нарративный вес ≠ экономический эффект [K-141]

### Card: `CausalModel` (эпистемический артефакт)
- **Definition:** формализованная закономерность с идентифицированным механизмом [K-097]
- **Attributes:** причина; механизм; промежуточные эффекты; следствия; ограничения; исключения; **cycle_annotation (обязателен при касании петли K-149)** [K-150]
- **Validation Rules:** без механизма → понижение до EmpiricalObservation [K-097]; пересмотр только при множественных независимых расхождениях [K-158]

## II.6. Модели процессов

**P-1. Цикл накопления/распределения** [K-089]: LTH накапливают (низкая цена/неопределённость) → доступный float ↓ → чувствительность цены к спросу ↑ → пик цикла → LTH распределяют к новым STH → волатильность ↑ → капитуляция STH → возврат к накоплению. *Аннотация:* один проход петли hashrate-цикла не включает — петли независимы [K-150].

**P-2. Halving-цикл** [K-092, K-085]: эмиссия ÷2 → доход майнеров в BTC ↓ → давление на маржинальных майнеров → продажное давление предложения ↓ (при прочих равных) → историческое давление на цену через 6–18 мес. *Ограничение:* институциональная эпоха может ослаблять (K-123, Q-02). *Аннотация цикла:* затрагивает петлю hashrate↔цена — описан один проход [K-150].

**P-3. Институциональный канал** [K-093, K-090]: регуляторное одобрение → барьер входа ↓ → CapitalFlow(institutional, inflow) → float ↓ + сдвиг профиля держателей. *Обратимость:* канал двунаправлен при смене макроцикла [K-093].

# Часть VI. Recommended Refactoring of the Original BDKS

Рефакторинг **сохраняет всё знание** (K-001…K-161 без потерь) и меняет только инженерную структуру:

**VI.1. Новая структура разделов (v1.1):**

```
0. Мета: версия, as_of-конвенция, владения (таксономия сигналов, Golden Dataset)
1. Границы и принципы            (Р1, без изменений содержания)
2. Controlled Vocabulary          (Р2 → нормализованный: канонические имена II.1,
                                   каждый термин с K-ID и кросс-ссылками вместо повторов)
3. Онтология                      (Р3 → верхняя иерархия II.2 + карточки II.4;
                                   +CentralBank, +Jurisdiction, +MacroEnvironment,
                                   +FeeMarket, +Price, +NarrativePhenomenon)
4. Таксономия событий             (Р4 + типология SecurityIncident)
5. Таксономия сигналов            (Р5 + маркер «единственный источник истины»)
6. Агенты                         (Р6, переименован из «участников», is_a к EconomicAgent)
7. Механизмы и процессы           (Р7 + процессные модели II.6)
8. Причинные модели               (Р8 + cycle_annotation + граф II.8 + правка G-12)
9. Метрики                        (Р9, каждая метрика → measured_by-связь с классом)
10. Источники                     (Р10 + расщепление достоверности R-11 в шкале)
11. Неопределённость              (Р11 без изменений)
12. Искажения                     (Р12 без изменений)
13. Прецеденты и Golden Dataset   (Р13 + декларация владения)
14. Карта знаний                  (Р14 + полная проекция классов на уровни)
15. Правила вывода                (НОВЫЙ: каталог R-01…R-12 — извлечён из прозы в явный вид)
16. Использование производными    (Р15 + версионный пиннинг)
17. Эволюция знаний               (Финальный + процедура перекалибровки)
Приложение A. Traceability Matrix (II.10, полная)
Приложение B. Machine-readable companion (план)
```

**VI.2. Правила миграции:** ни одно утверждение не удаляется; перемещение фиксируется в матрице «Р(v1.0)→§(v1.1)»; GAP-FILL элементы входят с пометкой источника (это ревью, не доменная литература) и подлежат содержательному наполнению только из материалов пользователя — в соответствии с ограничением Шага 1 алгоритма проекта.

**VI.3. Критерий приёмки v1.1:** все High-находки закрыты; R-02 (трассировка к принципам) выполняется для 100% классов; двойных определений ноль (каждое понятие определено в одном месте, остальные — ссылки); прогон governance-правила K-161 — прецеденты Р13 не потребовали переписывания под новую структуру (регресса знания нет).
