### Группировка по критичности

#### КРИТИЧЕСКИЕ (блокируют масштабирование)

| # | Проблема | Механизм возникновения |
|---|----------|----------------------|
| C1 | Бизнес-логика в клиентском коде | `synthesizeNarrativeAdvanced()` в `index.html` — дублируется в Python, невозможно тестировать изолированно |

| # | Проблема | Механизм возникновения |
|---|----------|----------------------|
| H1 | Дрейф знаний | `synthesis.json` устаревает; нет автоматической инвалидации при новых сигналах |
| H2 | Temporal blindness | Старый сигнал с высоким `contradicts` определяет нарратив даже после потери актуальности |

| H5 | Отсутствие `confidence` и `source_url` | Невозможна верификация; ошибочные интерпретации воспроизводятся с видимостью авторитетности |

| # | Проблема |
|---|----------|
| M1 | Необратимое информационное сжатие 840:1 без drill-down |
| M2 | Архив без семантики — исторические паттерны физически недоступны |
| M3 | Нет метрик качества — деградация незаметна |

### Причинно-следственные связи между рисками

```
C2 (JSON-монолит)
    ├── → H2 (temporal blindness — нет индекса по дате)
    └── → M2 (архив без семантики — нет структуры для запросов)

C1 (логика в клиенте)
    ├── → C3 (два источника истины — Python дублирует JS)
    └── → M3 (нет метрик — логику нельзя тестировать изолированно)

C4 (5 статичных кластеров)
    ├── → H3 (межкластерная слепота)
    └── → M4 (эхо-камера — нет механизма обнаружения новых паттернов)

H1 (дрейф знаний)
    └── → C5 (нет audit trail — нет истории версий синтеза)

H4 (единственная точка отказа)
    └── → M4 (эхо-камера — нет внешней проверки)
```

## Этап 2. Целевая архитектура

### Принципы

1. **Single Source of Truth** — каждое знание хранится в одном месте
2. **Event Sourcing** — история неизменяема, состояние вычисляется
3. **Separation of Concerns** — хранение, логика, отображение — разные слои
4. **Explainability First** — каждый вывод сопровождается цепочкой рассуждений
5. **Graceful Degradation** — система работает при частичной недоступности компонентов

### Схема целевой архитектуры

```
┌─────────────────────────────────────────────────────────────────────┐
│  СЛОЙ ВВОДА                                                         │
│  Signal Ingestion → Validation → Normalization                      │
└────────────────────────────┬────────────────────────────────────────┘
                             ↓
┌─────────────────────────────────────────────────────────────────────┐
│  СЛОЙ ЗНАНИЙ (Knowledge Layer)                                      │
│  Signal Store → Entity Store → Relationship Graph → Ontology        │
└────────────────────────────┬────────────────────────────────────────┘
                             ↓
┌─────────────────────────────────────────────────────────────────────┐
│  СЛОЙ АНАЛИЗА (Narrative Engine)                                    │
│  Scoring → Contradiction → Synthesis → Reasoning → Approval        │
└────────────────────────────┬────────────────────────────────────────┘
                             ↓
┌─────────────────────────────────────────────────────────────────────┐
│  СЛОЙ ДОСТАВКИ                                                      │
│  API → Cache → UI                                                   │
└────────────────────────────┬────────────────────────────────────────┘
                             ↓
┌─────────────────────────────────────────────────────────────────────┐
│  СЛОЙ ИСТОРИИ                                                       │
│  Archive → History → Analytics                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### Описание компонентов

---

#### 2.1 Signal Ingestion

**Назначение:** приём сигналов от аналитика, первичная валидация  
**Входные данные:** сырой объект сигнала от аналитика  
**Выходные данные:** нормализованный объект → Signal Store  
**Зависимости:** Ontology (для валидации кластера), Entity Store (для связей)  
**Жизненный цикл:** stateless; каждый вызов независим  
**Контракт:**
```python
def ingest_signal(raw: dict) -> ValidationResult:
    # Возвращает либо нормализованный сигнал либо список ошибок
    # Никогда не записывает в хранилище — только валидирует
```

---

#### 2.2 Signal Store (signals.json → signals_store/)

**Назначение:** единственное хранилище сырых сигналов  
**Структура:** append-only; сигналы никогда не удаляются, только помечаются `status: archived`  
**Индексы:** по `cluster`, `date`, `narrative_role`, `weight`  
**Жизненный цикл:** вечный; архивирование = смена статуса, не удаление  
**Целостность:** SHA-256 хэш каждого сигнала при записи  
**Контракт:**
```python
def get_signals(
    cluster: str | None,
    date_from: date | None,
    date_to: date | None,
    roles: list[str] | None,
    window_days: int = 90,
    include_archived: bool = False
) -> list[Signal]
```

---

#### 2.3 Entity Store (ENTITIES.json → entities/)

**Назначение:** каталог сущностей экосистемы Bitcoin  
**Входные данные:** новые сущности при обработке сигналов  
**Выходные данные:** обогащённые сигналы с entity-ссылками  
**Расширение:** добавить `aliases` (список альтернативных имён), `external_ids` (CoinGecko, Arkham), `confidence`  

---

#### 2.4 Ontology

**Назначение:** формальное определение кластеров, ролей, весов, связей  
**Ключевое изменение:** кластеры становятся **динамическими**  
**Хранение:** `ontology.json`

```json
{
  "version": "1.0",
  "clusters": {
    "strategy_model_stress": {
      "label": "STRATEGY: МОДЕЛЬ ПОД ДАВЛЕНИЕМ",
      "description": "...",
      "parent": null,
      "created": "2026-01-01",
      "deprecated": null,
      "signal_count": 11
    }
  },
  "roles": ["trigger", "complication", "resolution", "background"],
  "weights": ["onchain", "primary", "market", "media"],
  "scoring_rules": {
    "freshness": {"7": 3, "30": 1, "inf": 0},
    "weight": {"onchain": 4, "primary": 3, "market": 2, "media": 1},
    "role": {"trigger": 4, "complication": 3, "resolution": 2, "background": 0}
  }
}
```

**Создание нового кластера:** только через явную операцию с обоснованием  
**Депрекация кластера:** `deprecated` дата + `successor` кластер

---

#### 2.5 Relationship Graph

**Назначение:** хранение и вычисление связей между сигналами  
**Типы связей:** `confirms`, `contradicts`, `context_chain`  
**Направленность:** двунаправленная с разными семантиками  
**Хранение:** `relationships.json` — отдельно от сигналов

```json
{
  "edges": [
    {
      "from": "STR-2026-0625-001",
      "to": "STR-2026-0615-001",
      "type": "contradicts",
      "rationale": "macro_impl A: накопление без ограничений vs macro_impl B: накопление остановлено",
      "created": "2026-06-28",
      "created_by": "analyst"
    }
  ]
}
```

**Ключевое решение:** связи хранятся отдельно от сигналов → изменение связи не меняет сигнал → история связей сохраняется.

---

#### 2.6 Scoring Engine

**Назначение:** вычисление score для кластеров и отдельных сигналов  
**Реализация:** Python-библиотека `scoring.py` — единственная реализация  
**JS-версия:** тонкая обёртка вызывающая предварительно вычисленный результат  
**Входные данные:** список сигналов + правила из Ontology  
**Выходные данные:** `ClusterScore` объект с breakdown по осям

```python
@dataclass
class ClusterScore:
    total: int
    freshness: int
    weight: int
    tension: int
    roles: int
    strength: Literal["structural", "strong", "moderate", "weak"]
    signals_contributing: list[str]  # id сигналов которые внесли вклад
    computed_at: datetime
```

---

#### 2.7 Contradiction Engine

**Назначение:** автоматическое обнаружение противоречий между сигналами  
**Входные данные:** новый сигнал + существующие сигналы кластера  
**Выходные данные:** список кандидатов на `contradicts` с объяснением  
**Алгоритм:**
1. Семантическое сравнение `macro_implication` через keyword overlap
2. Проверка инверсии ключевых терминов (рост/падение, накопление/продажа)
3. Проверка временнóго конфликта (два сигнала про одно и то же событие с противоположными выводами)

**Выход:** предложение аналитику, не автоматическая запись  
**Правило:** система предлагает, человек решает

---

#### 2.8 Synthesis Engine (synthesizer.py)

**Назначение:** единственная реализация алгоритма синтеза нарратива  
**Язык:** Python (не JS)  
**Входные данные:** сигналы кластера + Relationship Graph + Ontology  
**Выходные данные:** `SynthesisResult`

```python
@dataclass
class SynthesisResult:
    cluster: str
    tension: str
    narrative: str        # partA + bridge + partB
    takeaway: str
    strength: str
    confidence: float     # 0.0–1.0
    causal_chain: list[str]
    signals_used: list[str]
    signals_ignored: list[str]   # НОВОЕ: что было проигнорировано
    rationale: str               # НОВОЕ: почему выбраны именно эти сигналы
    algorithm_version: str       # НОВОЕ: версия алгоритма
    computed_at: datetime
    window_days: int
```

**JS-версия:** читает `synthesis_cache.json` — не вычисляет  
**Тестируемость:** полностью покрыт unit-тестами

---

#### 2.9 Approval Engine

**Назначение:** workflow утверждения синтеза аналитиком  
**Состояния синтеза:**

```
draft → reviewed → approved → published → superseded → archived
```

**Хранение:** `synthesis_store/`

```
synthesis_store/
  2026-06-28-strategy_model_stress-v1.json
  2026-06-28-strategy_model_stress-v2.json  ← утверждённая версия
  2026-07-05-strategy_model_stress-v1.json  ← следующее обновление
```

**Каждый файл содержит:**
```json
{
  "id": "syn-strategy_model_stress-20260628-002",
  "cluster": "strategy_model_stress",
  "status": "approved",
  "version": 2,
  "content": {
    "tension": "...",
    "narrative": "...",
    "takeaway": "...",
    "strength": "structural"
  },
  "rationale": "Выбран STR-2026-0628-001 как complication т.к. primary weight и свежее",
  "alternatives_rejected": ["...альтернативная формулировка tension..."],
  "signals_used": ["STR-2026-0622-003", "STR-2026-0628-001"],
  "signals_window": 90,
  "algorithm_version": "2.1",
  "computed_draft": "2026-06-28T14:00:00Z",
  "approved_at": "2026-06-28T15:30:00Z",
  "approved_by": "analyst",
  "expires_at": "2026-07-05T00:00:00Z",
  "superseded_by": null
}
```

---

#### 2.10 Synthesis Cache (synthesis_cache.json)

**Назначение:** быстрый доступ для UI — один файл с актуальными утверждёнными синтезами  
**Обновляется:** автоматически при утверждении нового синтеза  
**Содержит:** только последний approved синтез каждого кластера  
**Срок жизни:** файл всегда актуален (нет TTL) — устаревание определяется полем `expires_at`

```json
{
  "meta": {
    "updated": "2026-06-28T15:30:00Z",
    "algorithm_version": "2.1",
    "signals_count": 42
  },
  "clusters": {
    "strategy_model_stress": {
      "synthesis_id": "syn-strategy_model_stress-20260628-002",
      "status": "approved",
      "expires_at": "2026-07-05T00:00:00Z",
      "content": { "tension": "...", "narrative": "...", "takeaway": "...", "strength": "structural" },
      "source": "approved"
    }
  }
}
```

---

#### 2.11 Fallback Engine (JS)

**Назначение:** вычислить синтез в браузере если `synthesis_cache.json` устарел  
**Условие активации:** `expires_at` прошёл ИЛИ файл недоступен  
**Индикация в UI:** явная метка «Алгоритмический черновик» — не скрытый fallback  
**Алгоритм:** упрощённая версия `synthesizer.py` — только финальный шаг, не полный пересчёт

---

#### 2.12 History Store

**Назначение:** неизменяемая история всех синтезов  
**Хранение:** `synthesis_store/` (все файлы, не только актуальные)  
**Запросы:** «как выглядел нарратив 30 января 2026?» → найти approved синтез на эту дату  
**Аналитика:** динамика изменения tension кластера за период

## Этап 3. Архитектурные решения (ADR)

---

### ADR-001: Вынос синтеза из index.html в synthesizer.py

**Проблема:** `synthesizeNarrativeAdvanced()` существует в двух реализациях (JS и Python) которые расходятся  
**Контекст:** любое изменение алгоритма требует синхронного обновления двух файлов на двух языках  
**Варианты:**
- A) Оставить в JS, добавить тесты → не устраняет дублирование
- B) Перенести в Python, JS читает результат → устраняет дублирование
- C) WebAssembly — одна реализация на Rust → избыточная сложность

**Выбор:** B — Python как единственная реализация  
**Преимущества:** тестируемость, единственный источник правды для алгоритма  
**Недостатки:** нужен шаг сборки при каждом изменении алгоритма  
**Влияние:** JS становится тонким клиентом; `synthesis_cache.json` — интерфейс между слоями

---

### ADR-002: Разделение signals.json на Signal Store + Relationship Graph

**Проблема:** `links.contradicts` внутри сигнала смешивает данные события (что произошло) с аналитическими связями (как это связано с другим)  
**Контекст:** изменение связи сейчас означает изменение самого сигнала — нарушение immutability  
**Выбор:** `relationships.json` как отдельный граф  
**Преимущества:** история связей независима от истории сигналов; можно менять интерпретацию не меняя факт  
**Недостатки:** два файла вместо одного; нужна синхронизация при добавлении  
**Влияние:** CLAUDE.md должен описывать два отдельных шага: добавление сигнала и построение связей

---

### ADR-003: Динамические кластеры через Ontology

**Проблема:** 5 статичных кластеров захардкожены в коде; новые явления не имеют места  
**Контекст:** Bitcoin-экосистема за 5 лет породит нарративы которых нет сейчас (CBDC-конкуренция, Layer 3, квантовые угрозы)  
**Выбор:** `ontology.json` как единственный источник определений кластеров  
**Преимущества:** добавление кластера = одна строка в JSON; история кластеров сохраняется  
**Недостатки:** нужна валидация при добавлении сигнала (кластер существует?)  
**Влияние:** все компоненты читают кластеры из Ontology, не из кода

---

### ADR-004: Versioned Synthesis Store

**Проблема:** `synthesis.json` перезаписывается при обновлении — история утерта  
**Контекст:** невозможно ответить на вопрос «какой был нарратив месяц назад?»  
**Выбор:** append-only директория `synthesis_store/` + `synthesis_cache.json` для UI  
**Преимущества:** полная история, rollback в один шаг, audit trail  
**Недостатки:** растёт объём хранилища  
**Влияние:** `synthesis_cache.json` генерируется автоматически из store

---

### ADR-005: Явные режимы в UI (Approved vs Draft)

**Проблема:** пользователь не знает смотрит ли он утверждённый анализ или алгоритмический черновик  
**Выбор:** явная метка в каждой карточке нарратива  
**Утверждённый:** `✓ Аналитик · 28 июня`  
**Черновик:** `◈ Алгоритм · черновик`  
**Просроченный:** `⚠ Обновить · 5+ дней`

---

### ADR-006: Разделение `theme` и `cluster`

**Проблема:** `theme` и `cluster` дублируют друг друга с незначительными отличиями  
**Выбор:** оставить оба поля но формализовать семантику:
- `theme` — тематическая категория из фиксированного набора (supply, institutionalization, infrastructure, macro, narrative)
- `cluster` — конкретный нарративный кластер (может быть несколько на одну тему)
**Влияние:** фильтрация в UI по theme; синтез — по cluster

---

#### 2.4 Contradiction Detector

**Что делает:** при добавлении нового сигнала анализирует существующие и предлагает кандидатов на `contradicts`  
**Алгоритм:**
```python
def detect_contradictions(new_signal: Signal, existing: list[Signal]) -> list[ContraCandidate]:
    candidates = []
    for sig in existing:
        if sig.cluster != new_signal.cluster:
            continue
        # Критерий 1: Инверсия ключевых терминов
        score = semantic_inverse_score(new_signal.macro_implication, sig.macro_implication)
        # Критерий 2: Конфликт actor + противоположный dir
        if new_signal.actor == sig.actor and new_signal.dir != sig.dir:
            score += 0.3
        # Критерий 3: Перекрытие по дате (про один период)
        if date_overlap(new_signal.date, sig.date, days=30):
            score += 0.2
        if score > 0.6:
            candidates.append(ContraCandidate(sig.id, score,
                reason=explain_contradiction(new_signal, sig)))
    return sorted(candidates, key=lambda c: -c.score)[:3]
```

**Выход:** предложение аналитику в виде:
```
Возможные противоречия (предложение — не автоматическая запись):
  STR-2026-0615-001 (score: 0.82) — «покупки без ограничений» vs «покупки остановлены»
  → Добавить в contradicts? [y/n]
```

```

      │
      ▼

      │
      ▼

      │

      ▼

      │
      ▼
[synthesis_cache.json] — перезапись
  Хранит: только последний approved синтез каждого кластера

  TTL: нет (устаревание = expires_at поле)
      │
      ▼
[index.html / UI]
  Читает: synthesis_cache.json
  Fallback: JS-алгоритм если expires_at прошёл
  Показывает: метку источника (Аналитик / Алгоритм)
      │
      ▼
[АРХИВ — при status: archived в signals.json]
  Сигналы старше 180 дней получают status: archived
  Из окна синтеза выпадают
  Из базы НЕ удаляются — доступны для исторических запросов
      │
      ▼
[ИСТОРИЯ]
  synthesis_store/ содержит все версии навсегда
  Запрос: «нарратив на 1 января 2026» → найти approved синтез на эту дату
```

### UI Paging Strategy (IRP v1 Wave 4 / MC07, 2026-07-02)

> Документация решения, не реализация. Triggers: RR-05 (IRP_v1.md) —
> внедрять при 300+ сигналов; исходная формулировка SCL06 (IRR) говорит
> «500+» — используется более консервативный порог 300 (раньше
> предупредить дешевле, чем упустить момент). На 2026-07-02: 49
> сигналов — далеко от обоих порогов, поэтому запланировано, не
> реализовано (мониторинг роста — `check_signals_size.py`, OP06).

**Где реальное узкое место.** `index.html::renderSignals()` не пагинирует
ничего сейчас — `SIGNALS` (весь массив из `fetch('signals.json')`)
фильтруется в `items`, затем **весь** `items.map(cardHTML).join('')`
строится и вставляется в DOM за один вызов. Каждая карточка (`cardHTML()`)
— это несколько `sanitize()`/`highlightEntities()` вызовов, chips, два
параграфа текста. Проблема не в загрузке данных (`signals.json` при
300-500 сигналах — не больше десятков КБ, `fetch` тривиален), а именно
в объёме DOM и объёме строкового рендера за один синхронный проход.

**Решение — пагинировать рендер, не данные.** `SIGNALS` остаётся полным
массивом в памяти (нет backend, нечего постранично запрашивать —
источник по-прежнему один статический `signals.json`). Пагинация
применяется к уже отфильтрованному `items`, **после** `sigFilter`/
`sigActorFilter`, **до** `cardHTML().join('')`:

```
SIGNALS (весь массив, как сейчас)
  → filter (sigFilter, sigActorFilter)   — без изменений
  → sort (по date, как сейчас)           — без изменений
  → items                                 — как сейчас
  → items.slice(0, sigPage * PAGE_SIZE)   — НОВОЕ: окно рендера
  → cardHTML().join('')
```

Паттерн — «Load more» (кнопка добавляет следующую партию), не
нумерованные страницы: сайт уже feed-подобный (лента с фильтрами
сверху), нумерация страниц добавила бы состояние (`currentPageNumber`)
без выигрыша для этого UX — «показать ещё» проще и достаточно.
`PAGE_SIZE` — предварительная оценка **40**, не финальное число (тот же
принцип, что у Rate Limits в API.md — пересмотреть на реальных
замерах когда порог будет близко, не гадать сейчас).

**Открытый вопрос для реализации (сознательно не решается здесь):**
режим `theme` группирует `items` по теме перед рендером — простой
`slice()` по общему списку не транслируется чисто на группы (не ясно,
резать ли количество групп или количество карточек внутри каждой).
Практически это может не быть проблемой: сигналы делятся между ~5-7
темами, то есть при 300 сигналах в среднем ~40-60 на тему — уже
терпимый объём на группу без пагинации. Решить эмпирически при
подходе к порогу 300, не проектировать заранее для гипотетического
распределения.

**Не входит в этот документ:** сам код (`sigPage` state, кнопка
«Показать ещё», срез `items`) — появится отдельным PR при подходе к
порогу 300, через `docs/spec-pilot.md` (правило CLAUDE.md: изменения
структуры/поведения `index.html` требуют чтения этого алгоритма перед
правкой).

- **Создать validator.py** — без валидации данные деградируют незаметно
- **Создать relationships.json** — изменение связей сейчас = изменение сигнала (нарушение immutability)

### Допустимо принять сейчас

- `cat` + `catLabel` дублирование → устранить при следующем рефакторинге схемы
- Отсутствие `confidence` поля → добавить как optional в Фазе 1
- Бизнес-логика в `index.html` (часть) → постепенная миграция через Фазы 0-1

### Можно отложить до 200+ сигналов

- Разбивка `signals.json` на файлы по году
- Backend вместо GitHub Pages
- Semantic search

### Категорически нельзя откладывать

- **Audit trail** — каждый день без него = день истории потерян навсегда
- **Правило contradicts** — уже нарушалось; повторение = деградация Relationship Graph

## Итоговый вердикт

Текущая система — **хорошо спроектированный MVP**. Blueprint не предлагает революцию. Он предлагает эволюцию: устранить три фундаментальных проблемы (дублирование логики, отсутствие audit trail, два источника истины) и заложить фундамент для масштабирования.

**Главный принцип реализации:** каждая Фаза должна быть завершена и верифицирована до начала следующей. Не начинать Фазу 1 пока `synthesizer.py` не доказал идентичность с JS-версией.

**Стоимость ошибки откладывания:** каждый день без audit trail = день истории синтезов потерян навсегда. Это единственный необратимый риск в текущей системе.
