## Часть I. Назначение и границы

### Назначение

Narrative Intelligence Engine (NIE) — логическая система, преобразующая разрозненный поток наблюдений о Bitcoin в связный, объяснимый и проверяемый корпус аналитического знания: сигналы, их связи, кластеры противоречий и синтезированные нарративы.

NIE операционализирует методологию BAMS над знанием BDKS. Он не генерирует знание из ничего и не заменяет аналитика — он гарантирует, что каждый вывод прошёл через обязательные методологические шаги и остался прослеживаемым до исходных наблюдений.

### Что система делает

- Принимает материал о событии, отклоняет непроверяемое (BAMS Р1: сигнал без источника — слух);
- Проводит верификацию через независимые источники (BAMS Р6);
- Классифицирует наблюдение по таксономиям BDKS (Р4, Р5) и связывает с онтологией (Р3);
- Устанавливает связи с существующим корпусом: подтверждение, противоречие, контекст (BAMS Р7);
- Синтезирует интерпретационные конструкции: tension, macro_implication, роль в нарративе (BAMS Р8);
- Показывает рассуждение до фиксации результата (BAMS Р11 — принцип видимости);
- Ведёт производные базы сущностей и механизмов, прослеживаемые до сигналов;
- Собирает из корпуса сигналов нарративные карточки детерминированным алгоритмом выбора (ALGORITHM.md).

### Что система не делает

- Не прогнозирует цену (BDKS Р3, класс Price; BAMS Р14);
- Не анализирует объекты вне причинной связи с Bitcoin (BDKS Р1, правило R-10);
- Не выдаёт вывод без прослеживаемости до сигналов (BAMS Финальный раздел);
- Не заполняет пробелы данных обобщённым знанием модели (инвариант «отсутствие скрытого домысливания»).

## Часть II. Архитектурные принципы

| Принцип | Воплощение в NIE | Источник требования |
|---|---|---|
| Single Source of Truth | Домен — только BDKS; метод — только BAMS; корпус сигналов — единственная база записей; производные базы ссылаются, не копируют | BDKS Р15, R-03 |
| Explainability | Интерпретационные решения показываются до фиксации; объяснение — не приложение к выводу, а способ его сборки | BAMS Р11 |
| Traceability | Каждый вывод → сигналы; каждый сигнал → источник; каждая запись производной базы → сигналы; id неизменяемы | BAMS Финальный; CLAUDE.md (правило неизменности id) |
| Determinism | Метод применяется одинаково к одинаковому входу; пакетный синтез нарративов — чистая функция от корпуса | BAMS Финальный (детерминированность метода) |
| Modularity | Подсистемы с непересекающимися зонами ответственности (Часть V) | Настоящий документ |
| Separation of Concerns | Три уровня: знание / метод / реализация — независимая эволюция каждого | BDKS Преамбула |
| Technology Independence | NIES не ссылается на языки, форматы файлов, конкретные хранилища | Настоящий документ |
| Extensibility | Открытые таксономии (кластеры, типы механизмов) с явной процедурой расширения | BDKS Финальный раздел |
| Reproducibility | Повторный прогон метода на том же корпусе даёт тот же результат | BAMS Р1 (воспроизводимость) |
| Auditability | История изменений знания, метода и корпуса ведётся с обоснованиями | BDKS Финальный, R-07 |
| Event Sourcing | Каждый значимый переход состояния (сигнал добавлен/архивирован, синтез утверждён/устарел, связь отозвана) фиксируется как неизменяемая запись в добавочном журнале, отдельно от самого состояния; реакции системы на переход (инвалидация производных представлений) — явные обработчики поверх журнала, не побочный эффект внутри операции записи | Blueprint Этап 2, Принцип 2 «Event Sourcing»; реализация — `domain/events.py` + `domain/lifecycle.py` |

## Часть III. Контекст системы

### Внешние акторы

| Актор | Роль во взаимодействии |
|---|---|
| **Оператор-аналитик** (владелец домена + заказчик) | Поставляет материал; подтверждает интерпретационные решения; принимает изменения знания и метода (BAMS: модель ответственности) |
| **Исполнитель метода** | Человек или система, ведущая сигнал через пайплайн; в текущей реализации роли исполнителя и движка совмещены |
| **Внешние источники наблюдений** | Первоисточники и вторичные источники по типам сигналов (BDKS Р10 — шкала надёжности) |
| **Читатель** | Потребитель синтезированных нарративов и корпуса; не влияет на содержание |

### Входы системы (граница)

- Материал о событии: текст, ссылка, изображение — в любом виде, пригодном для фиксации «что именно поступило»;
- Результаты верификации из внешних источников;
- Решения оператора: подтверждение/корректировка интерпретаций, приёмка изменений.

### Выходы системы (граница)

- Записанный сигнал (структурированная запись + читаемый архив);
- Сводка решения (компактное резюме уже показанного рассуждения);
- Обновления производных баз (сущности, доходные механизмы);
- Нарративные карточки (пакетный синтез);
- Записи аудита: версии, обоснования изменений.

## Часть IV. Логическая архитектура

### Обзор подсистем

```
                                ┌─────────────────────┐
 Материал ──► S1 Приём ──► S2 Верификация ──► S3 Разбор │  интерактивный
                │              │                 │      │  контур
                ▼              ▼                 ▼      │  (сигнал)
              [отказ]    S10 Доступ к знанию (BDKS/мосты)
                               │                 │
                        S4 Классификация ◄───────┘
                               │
                        S5 Связывание ◄──── Корпус сигналов
                               │
                        S6 Синтез интерпретаций
                               │
              S7 Объяснение (видимость) ──► Оператор ──► подтверждение
                               │
                        S8 Валидация
                               │
                        S9 Фиксация и трассировка ──► Корпус + производные базы
                                                          │
                              пакетный контур:            ▼
                        S11 Нарративный композитор ──► Карточки нарративов
                               ▲
                        S12 Управление изменениями (governance, поверх всего)
```

Два контура работы: **интерактивный** (обработка одного сигнала с человеком в петле, S1–S9) и **пакетный** (детерминированная сборка нарративов из корпуса, S11). S10 и S12 — сквозные сервисы.

## Часть V. Спецификации подсистем

Формат: назначение · ответственность · входы → выходы · зависимости · отказ · качество.

### S1 — Приём материала (Input Management)

- **Назначение:** зафиксировать поступившее наблюдение без интерпретации.
- **Ответственность:** определить форму материала; зафиксировать исходную точку; отклонить материал вне границ домена (RQ-17) и материал без перспективы верификации (RQ-01). Не наполнять наблюдение знанием системы — только материалом источника.
- **Входы → выходы:** сырой материал → зафиксированное наблюдение ∨ мотивированный отказ.
- **Зависимости:** S10 (проверка границ домена по BDKS Р1).
- **Отказ:** материал вне домена; материал-слух. Отказ — валидный исход с объяснением.
- **Качество:** нулевая интерпретация на этом шаге; фиксация дословна.

### S2 — Верификация (Research & Source Verification)

- **Назначение:** установить достоверность наблюдения через независимые источники.
- **Ответственность:** найти первоисточник; привлечь дополнительные источники по типу сигнала (карта источников — BDKS Р10); проверить независимость (RQ-03); зафиксировать расхождения; определить дату события по приоритету первоисточника, не по формулировкам оператора.
- **Входы → выходы:** зафиксированное наблюдение → верифицированное наблюдение с картой источников и расхождений.
- **Зависимости:** внешние источники; S10 (шкала надёжности).
- **Отказ:** первоисточник недоступен → работа продолжается с явной пометкой; источники противоречат в ключевом факте → расхождение обязано попасть в ограничения (RQ-10).
- **Качество:** для on-chain — раздельная оценка факта и атрибуции (RQ-05).

### S3 — Разбор (Signal Decomposition)

- **Назначение:** структурировать верифицированное наблюдение в читаемый промежуточный артефакт.
- **Ответственность:** четыре грани разбора — что это / контекст / что означает сейчас / оговорки. Здесь же — обязательное рассмотрение альтернативных объяснений (RQ-06): выбранная интерпретация и хотя бы одна отклонённая.
- **Входы → выходы:** верифицированное наблюдение → интерпретационный разбор.
- **Зависимости:** S10 (онтология задаёт пространство объяснений — BAMS Р5).
- **Отказ:** невозможность интерпретации при текущих данных → явное признание недостаточности (RQ-16).
- **Качество:** каждая грань разбора питает конкретные поля записи (детерминированный маппинг — уровень реализации).

### S4 — Классификация и кластеризация (Classification & Clustering)

- **Назначение:** поместить наблюдение в таксономии и кластеры корпуса.
- **Ответственность:** тема, категория, актор, поток, горизонт, достоверность — строго по таксономиям BDKS через мост знание↔реализация; кластер — по двум признакам одновременно (тема + субъект действия); решение о новом кластере — по явному правилу порога.
- **Входы → выходы:** интерпретационный разбор → классифицированный сигнал.
- **Зависимости:** S10; корпус сигналов (существующие кластеры).
- **Отказ:** наблюдение не сопоставимо ни с одним классом онтологии → заявка на расширение BDKS через S12, не подгонка под ближайший класс (BAMS Р2).
- **Качество:** канонические метки без вариаций; классификация не использует конструкции синтеза (tension) как критерий.

### S5 — Связывание (Linking & Contradiction Manager)

- **Назначение:** установить связи нового сигнала с корпусом.
- **Ответственность:** отбор кандидатов (та же тема; тот же кластер); три независимых критерия — подтверждение (то же утверждение другим механизмом), противоречие (тест: могут ли оба быть верны одновременно — BAMS Р7), контекстная цепочка (без X сигнал теряет смысл). Пустая связь — норма. Запрет накрутки противоречий ради влияния на выбор нарратива. Связь имеет собственный жизненный цикл после создания: предложена → активна → *опционально* отозвана (с обоснованием отзыва) — отзыв применяется когда связь признана ошибочной постфактум (ложное противоречие, неверная атрибуция), не переписывается на другой тип и не удаляется физически (Часть VII, жизненный цикл связи).
- **Входы → выходы:** классифицированный сигнал + корпус → связанный сигнал.
- **Зависимости:** корпус сигналов; S10 (тест противоречия — канон BAMS Р7).
- **Отказ:** нет.
- **Качество:** каждая связь мотивирована пройденным критерием, показанным оператору (S7); отзыв связи мотивирован не слабее исходного создания.

### S6 — Синтез интерпретаций (Interpretation Synthesizer)

- **Назначение:** сформулировать интерпретационные конструкции сигнала.
- **Ответственность:** tension (конфликт двух механизмов, не факт — RQ-08); macro_implication (структурное изменение, не пересказ); нарративная роль (по критериям: триггер / усложнение / разрешение / фон).
- **Входы → выходы:** связанный сигнал → синтезированный сигнал.
- **Зависимости:** S5 завершён (роль зависит от связей).
- **Отказ:** конструкции не проходят критерии формы → возврат на соответствующий шаг.
- **Качество:** нарратив следует порядку факт → механизм → гипотеза → доказательства → альтернативы → вывод (BAMS Р8); уверенность и ограничения не растворяются в тоне.

### S7 — Объяснение (Explanation Engine)

- **Назначение:** сделать рассуждение видимым до фиксации.
- **Ответственность:** показать оператору интерпретационные решения S3–S6 с обоснованиями до записи; свёртка процесса в готовый результат — нарушение метода, не стиль подачи (BAMS Р11). После фиксации — компактная сводка уже показанных решений.
- **Входы → выходы:** решения S3–S6 → показанное рассуждение → подтверждение оператора.
- **Зависимости:** оператор в петле.
- **Отказ:** оператор корректирует → возврат на затронутый шаг.
- **Качество:** полный набор элементов объяснимости (цепочка, доказательства, отвергнутые альтернативы, обоснование выбора, уверенность, ограничения) — не подмножество по выбору.

### S8 — Валидация (Validation Engine)

- **Назначение:** проверить готовность записи до фиксации.
- **Ответственность:** структурная целостность записи; уникальность и формат идентификатора; соответствие меток канону; заполненность обязательных полей; целостность ссылок (все связи указывают на существующие записи; производные базы — на существующие сигналы и сущности).
- **Входы → выходы:** синтезированный сигнал → валидированный сигнал ∨ список дефектов.
- **Зависимости:** корпус (проверка ссылок); автоматизированный контур проверки при интеграции изменений.
- **Отказ:** дефект → возврат с указанием шага.
- **Качество:** валидация детерминирована и автоматизируема.

### S9 — Фиксация и трассировка (Persistence & Traceability Manager)

- **Назначение:** записать сигнал и поддержать прослеживаемость корпуса.
- **Ответственность:** атомарная запись в корпус (структурированная запись + читаемый архив — одно изменение); присвоение неизменяемого идентификатора; обновление производных баз (сущности — участники события; доходные механизмы — при относимости) с обратными ссылками; испускание события фиксации в добавочный журнал (Event Sourcing, Часть II) — журнал и корпус обновляются как одна логическая операция, реакции на событие (инвалидация производных представлений) — отдельные обработчики, не часть самой записи.
- **Входы → выходы:** валидированный сигнал → записанный сигнал + обновлённые производные базы + событие в журнале.
- **Зависимости:** S8 пройдена.
- **Отказ:** конфликт идентификатора → пересчёт номера, повторная валидация.
- **Качество:** после записи идентификатор не меняется никогда (RQ-11); расхождение префикса и уточнённой темы не исправляется задним числом.

### S10 — Доступ к знанию (Knowledge Access)

- **Назначение:** единая точка обращения к знанию домена и мостам.
- **Ответственность:** предоставлять таксономии, онтологию, карту источников, правила BDKS по ссылке; мост знание↔реализация (соответствие полей записей классам знания); запрет локальных переопределений (R-03).
- **Входы → выходы:** запрос подсистемы → каноническое знание + применимые правила.
- **Зависимости:** BDKS (версионированная опора), словарь данных.
- **Отказ:** знание отсутствует → заявка на эволюцию BDKS через S12, не локальная заплатка.
- **Качество:** фиксация версии знания, на которую опирается корпус.

### S11 — Нарративный композитор (Narrative Composer, пакетный контур)

- **Назначение:** собрать из корпуса карточки главных нарративов.
- **Ответственность:** детерминированное ранжирование сигналов кластера; выбор якорного сигнала; сборка карточки из уже написанных полей — композитор выбирает лучшее, не генерирует новое (RQ-18); правило смены главного противоречия кластера — только через честный тест связывания, не через накрутку. Ядро вычисления (собственно синтез карточки) архитектурно отделено от оркестрации ввода-вывода (загрузка корпуса и предыдущего результата, сохранение карточки, обработка ошибки одного кластера без остановки остальных) — это два разных обязательства внутри одной подсистемы, а не одно: смешение уже однажды приводило к нарушению чистоты вычисления и было исправлено (ALGORITHM.md §17).
- **Утверждение результата (человек в петле — уточнение v2.0):** вычисленная композитором карточка — кандидат, не автоматически публикуемый результат. Между вычислением и публикацией стоит отдельный шаг подтверждения оператором (сверка tension/narrative, ввод обоснования выбора), после которого кандидат становится утверждённым и попадает в представление для читателя; до этого шага прежний утверждённый результат остаётся действующим. Это не часть чистого вычисления S11 (которое остаётся детерминированной функцией от переданных значений) — это управляющий шаг поверх него, тем же логическим уровнем, что подтверждение оператора в S7 для интерактивного контура. Ранее (до v2.0) этот документ ошибочно описывал пакетный контур как не требующий участия оператора — исправлено по факту существования отдельного шага утверждения в реализации.
- **Входы → выходы:** ядро — корпус сигналов + предыдущий результат (как значение, не как чтение источника) → карточка-кандидат; шаг утверждения — карточка-кандидат + решение оператора → утверждённая карточка ∨ отклонение с возвратом к прежней утверждённой версии.
- **Зависимости:** корпус; спецификация алгоритма ранжирования — ALGORITHM.md (Reasoning-слой, вне NIES).
- **Отказ:** кластер без синтезированных конструкций → карточка не собирается, кластер помечается; сбой оркестрации одного кластера не останавливает обработку остальных; оператор отклоняет кандидата → прежняя утверждённая версия остаётся действующей, кандидат не публикуется.
- **Качество:** ядро — чистая функция от переданных значений (Часть II, Separation of Concerns) — повторный вызов с теми же значениями даёт тот же результат независимо от того, как эти значения были получены; шаг утверждения — не чистая функция (зависит от решения оператора), и не обязан быть.

### S12 — Управление изменениями (Governance Controller)

- **Назначение:** управлять эволюцией знания, метода и архитектуры.
- **Ответственность:** изменения знания — по процедуре BDKS (обоснование недостаточности прежней формулировки, R-07; порог пересмотра, R-08); изменения метода — с проверкой на исторических прецедентах до принятия (BAMS Р13); версионирование документов и инструкций с журналом; модель ответственности (владелец домена / исполнитель метода / реализация); защита целостности: изменения корпуса и производных баз проходят автоматизированную проверку до интеграции.
- **Входы → выходы:** предложение изменения → принятое изменение с обоснованием ∨ отклонение.
- **Зависимости:** все подсистемы (сквозной сервис).
- **Отказ:** изменение без обоснования не принимается; регресс на прецедентах — основание отклонения.
- **Качество:** направление синхронизации неизменно — производное приводится к источнику.

## Часть VII. Модель жизненного цикла

### Жизненный цикл сигнала

```
предложен → верифицирован → интерпретирован → классифицирован →
связан → синтезирован → показан → валидирован → записан → [неизменяемое ядро]
```

После записи неизменяемы: идентификатор, дата, исходный источник. Изменяемы через полный повторный проход соответствующих шагов: связи (при появлении новых сигналов корпуса). Синтезированные конструкции (`tension`, `macro_implication`) неизменяемы **с момента первого утверждённого пакетного синтеза, в котором сигнал использован как якорный или влияющий** — это правило Immutability Policy (Blueprint D13), формально принятое также на уровне CLAUDE.md v7.9 (AD-7 закрыт 2026-07-05 в пользу D13): смена главного противоречия кластера после этой точки достигается исключительно добавлением нового сигнала, не правкой существующего. Обратимость вывода — инвариант BAMS: новый сигнал способен ослабить существующий вывод.

**Формальные состояния (state machine) записи сигнала:** `предложен/черновик → активен → архивирован`, с петлёй `невалиден → (исправление) → черновик` при неуспехе валидации (S8). `Архивирован` — терминальное состояние, обратный переход запрещён необратимо (архивирование по возрасту или вручную не отменяется). Это конкретизация абстрактного пайплайна выше: «предложен…связан…валидирован» соответствуют переходу `черновик→активен` (или петле в `невалиден` при отказе S8); «записан» — вход в `активен`; последующая архивация — отдельный, более поздний переход `активен→архивирован`, не часть исходной обработки одного сигнала.

### Жизненный цикл кластера

```
предложен → активен → (при явном решении) устаревший → архивирован
```

Кластер создаётся при пороге общности противоречия (переход в `активен` — или отклоняется, `предложен→архивирован` минуя активность, если решение отрицательное); главный tension активного кластера определяется композитором по правилу ранжирования (Reasoning-слой). Переход в `устаревший` требует явного решения с указанием кластера-преемника (если есть) и не меняет уже записанные сигналы устаревшего кластера — они остаются с исходным `cluster` (правило неизменности, S9). `Устаревший` кластер не принимает новые сигналы с датой события позже даты устаревания; последующий переход в `архивирован` — только для чтения.

### Жизненный цикл связи

```
предложена → активна → (опционально) отозвана
```

Связь создаётся в S5 сразу активной (в отличие от сигнала и кластера, промежуточного состояния «предложена» как отдельного шага с самостоятельным подтверждением в текущей реализации нет — оно логически совпадает с моментом создания). Отзыв (`активна→отозвана`) — единственный способ аннулировать ошибочно установленную связь: связь не удаляется физически и не меняет тип на другой задним числом — отзыв мотивирован не слабее исходного критерия связывания (S5) и сам является прослеживаемым решением (Часть II, Traceability). `Отозвана` — терминальное состояние для этой конкретной связи; при необходимости связать те же два сигнала заново создаётся новая запись связи, не реанимируется старая.

### Жизненный цикл пакетного синтеза (уточнение v2.0)

```
вычислен (кандидат) → [опционально: на доработке] → утверждён → опубликован →
(при появлении новой утверждённой версии) заменён → архивирован
```

Отличие от жизненного цикла сигнала: здесь два независимых участника — детерминированное вычисление (S11, ядро) производит кандидата; оператор утверждает или отклоняет (S11, шаг утверждения, человек в петле — см. исправление в S11 выше). Отклонённый кандидат не входит в цепочку состояний утверждённых версий — прежняя утверждённая версия остаётся действующей до следующего успешно утверждённого пересчёта. `Заменён` и `архивирован` — переходы только вперёд; возврат из `заменён` в `утверждён` запрещён необратимо (нельзя «воскресить» замененную версию — при необходимости пересчитывается заново от текущего корпуса).

### Жизненный цикл записи производной базы

```
сущность/механизм упомянут как участник события → создание записи →
обновления при каждом относящемся сигнале (+обратная ссылка) → статус: активен/закрыт
```

Закрытие — состояние, не удаление: запись остаётся прослеживаемой.

### Жизненный цикл знания и метода

Управляется S12 по процедурам BDKS/BAMS: версия, обоснование, проверка на прецедентах, синхронизация производных. Мажорное изменение знания не применяется к производным автоматически (BDKS: обратная совместимость).

### Согласованность между жизненными циклами (без транзакций)

Сигнал, связь, кластер, синтез и производные базы — независимые записи без общей транзакции между ними (в реализации — независимые файлы). Согласованность обеспечивается не атомарностью нескольких записей одновременно, а тремя механизмами:

1. **Упорядоченные операции** — каждый пайплайн выполняется в фиксированном порядке (запись → событие в журнал → реакции на событие), и атомарна только каждая отдельная запись, не вся цепочка целиком; частичный сбой посреди цепочки не откатывает уже выполненные более ранние шаги.
2. **Согласованность в конечном счёте (eventual consistency)** — производные представления (в т.ч. пакетные карточки нарративов) могут на короткое, ограниченное окно отставать от корпуса после его изменения; окно закрывается очередным пересчётом, не бесконечно.
3. **Проверка при чтении, не только при записи** — целостность ссылок (все связи и обратные ссылки производных баз указывают на существующие записи) проверяется отдельным, периодическим проходом, независимо от того, что каждая отдельная запись прошла валидацию (S8) в момент своего создания.

Известная граница модели: если запись состояния успешна, а испускание события в журнал (Часть II, Event Sourcing) — нет, возникает разрыв между состоянием и его прослеживаемостью (журнал не отражает реально произошедшее изменение). Разрешение конфликта при расхождении: состояние корпуса — источник истины, производные представления пересчитываются от него, не наоборот (тот же принцип, что и в S12 — «направление синхронизации неизменно»).

### Режимы работы
