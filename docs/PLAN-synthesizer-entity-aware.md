# PLAN: Entity-aware усиления синтезатора — статус и план на будущее

> **Начато:** 2026-07-20 · **Обновлено:** 2026-07-21
> **Источник:** эксперимент на кластере `btc_treasury_competition` (21 сигнал) — см. `docs/ALGORITHM.md` для самого алгоритма синтеза, этот документ — только журнал находок и план их учёта.

**Принцип, общий для всего документа:** ничего не завязывать на имя конкретного кластера или сущности. Каждое изменение — на измеримом свойстве данных (число уникальных сущностей, доля сигналов с определённым признаком, соотношение ролей), чтобы работать на любом будущем кластере с похожей структурой, не только на том, где находка впервые обнаружена.

---

## Выполнено (реализовано и смёржено)

### Фаза A — entity-aware дедупликация ([PR #397](https://github.com/AlxCheh/Bitcoin-Intel/pull/397))

Ключ дедупликации `(date, actor, cluster, dir)` ложно схлопывал сигналы о РАЗНЫХ компаниях одного `actor='corporate'` в «дубликаты». Добавлен опциональный `signal_entity_map` ({signal_id: entity_id} из `ENTITIES.json`), уточняющий ключ до сущности; фолбэк на `actor` для сигналов без привязки к одной компании (агрегатные метрики).

**Проверено на проде:** `btc_treasury_competition` — 3 реальных сигнала перестали теряться (18→21 в синтезе).

### Фаза B — диагностика периферийности anchor ([PR #399](https://github.com/AlxCheh/Bitcoin-Intel/pull/399))

Новые additive-поля `entity_count` / `anchor_entity_share` / `is_minority_anchor` в `SynthesisResult`. Пороги — `MULTI_ENTITY_THRESHOLD=5`, `MINORITY_ANCHOR_SHARE=0.15` (config/settings.py). Только измерение — выбор tension/anchor не меняется.

**Проверено на проде:** `btc_treasury_competition` — `is_minority_anchor=True` (El Salvador/МВФ, 2 сигнала из 21).

### Фаза C — гранулярность phase ([PR #401](https://github.com/AlxCheh/Bitcoin-Intel/pull/401))

`_detect_phase()` при `trigger>0 И complication>0` всегда возвращала `"active"`, независимо от соотношения. Новая константа `COMPLICATION_DOMINANCE_RATIO=3` — `complication ≥ 3×trigger` даёт `"tension"`, даже при наличии trigger. Resolution по-прежнему побеждает первой безусловно.

**Побочная находка:** golden fixture (`tests/golden/expected/golden_synthesis.json`) поле `phase` нигде реально не проверяется существующими тестами — риск регрессии оказался ниже, чем предполагалось. Отдельная, не исправленная в рамках этой фазы находка.

**Проверено на проде:** ровно 1 из 7 живых кластеров сменил фазу (`btc_treasury_competition`: active→tension) — не два, как предполагалось до проверки (предварительная оценка не учла эффект Фазы A на другой кластер).

### Находка 1 — `contradicts_share` вместо бинарного `has_contradicts` ([PR #403](https://github.com/AlxCheh/Bitcoin-Intel/pull/403))

`has_contradicts: bool` в `calculate_confidence()` не различал кластер с 1 contradicts-связью из 5 и с 7 из 13 (реальный разброс 0%–54%). Заменён на `contradicts_share: float`, линейная интерполяция между уже принятыми границами (0.8 при 0%, 1.0 при 100%) — **не** пересчёт коэффициентов.

**Важно:** перед реализацией обнаружился прямой смежный вопрос с `docs/ADR-011-confidence-calibration-deferred.md` (та комиссия отложила калибровку коэффициентов формулы до 30 синтезов) — вопрос был явно задан пользователю, реализовано с явным обоснованием отличия (см. докстринг `calculate_confidence()` и заметку 2026-07-21 в самом ADR-011).

**Проверено на проде:** 5 из 7 кластеров показали реальное снижение confidence на 0.08–0.13 (кластеры с частичной, не 0% и не 100%, contradicts-связностью).

## Находка 2 — закрыта, план не требуется

Потолок confidence (contradicts-бонус не ограничен сверху при линейном знаменателе) — уже разобран `docs/ADR-011-confidence-calibration-deferred.md` (30 июня 2026, независимо от этих экспериментов). Решение принято: не калибровать коэффициенты до 30 синтезов, мониторинг — `scripts/quality_report.py` (`MIN_SYNTHESES_FOR_CALIBRATION=30`).

**Статус на 2026-07-21:** в `synthesis_store/` — 16 файлов, порог не пройден. Механизм работает штатно, отдельного плана не заводится — при достижении порога `quality_report.py` сам напомнит.

## Находка 3 — план на будущую реализацию (не реализовано)

### Проблема

Сущность `strategy` сейчас участвует одновременно в двух кластерах — `strategy_model_stress` и `bitcoin_governance_debate` (через `NAR-2026-0711-001`). Синтезатор обрабатывает каждый кластер независимо — общая централь­ность одного актора для двух разных нарративов сейчас нигде не фиксируется. **Пока единичный подтверждённый случай, не статистический паттерн.**

### Техническое решение (по аналогии с Фазой B — только измерение)

Новая функция уровня `main()` (не внутри `synthesize_cluster()` — контракт §17 не даёт кластеру знать о других кластерах, агрегация только снаружи, после того как все кластеры уже синтезированы):

```python
def _find_cross_cluster_entities(
    all_signals: list[dict],
    signal_entity_map: dict,
) -> dict[str, set[str]]:
    """entity_id -> множество кластеров, в которых у него есть сигналы."""
    entity_clusters: dict[str, set[str]] = {}
    for s in all_signals:
        eid = signal_entity_map.get(s.get("id", ""))
        if eid and s.get("cluster"):
            entity_clusters.setdefault(eid, set()).add(s["cluster"])
    return {e: cs for e, cs in entity_clusters.items() if len(cs) > 1}
```

Записывается отдельным top-level ключом `_cross_cluster_entities` в `data/synthesis_cache.json` (не внутри per-cluster записи — данные про два кластера сразу).

**Порог для отчёта** (измеримое свойство, не хардкод по имени сущности): сущность попадает в отчёт при **≥2 сигналах в КАЖДОМ** из ≥2 кластеров — отсекает разовые случайные упоминания.

### Что сознательно НЕ делать (по аналогии с Фазой B)

- Не менять контент/тональность карточек кластеров на основе этой находки.
- Не отображать на сайте в первой итерации — сначала накопить наблюдения по мере роста базы (сейчас 1 случай).

### Тестирование (при реализации)

- Синтетический тест: 2 сущности, одна — в одном кластере, другая — в двух с ≥2 сигналами в каждом → находит только вторую.
- Тест на реальных данных: подтвердить нахождение `strategy` (уже известный случай, не синтетика).
- Полный прогон + `validate_integrity.py`.

### Приоритет

Ниже находок A/B/C/1 — единичный случай, инфраструктура (`signal_entity_map`) уже существует, реализация — тонкая надстройка. Откладывание не создаёт риска в отличие от находки 1 (там снижение confidence было уже происходящей неточностью).
