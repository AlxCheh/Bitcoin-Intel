# SBBA Implementation Remediation Plan — SBBA-IRP v1.0

**Основание:** SBBA v1.0 (`docs/SBBA_v1.md` · 2026-07-06) — реестр из 6 расхождений (OG-1…OG-6)
**Цель:** довести реестр SBBA до состояния «закрыт / сознательно принят / ждёт решения владельца» — по аналогии с `IRP_v1.md`, но без объёма его 31 замечания: здесь 4 карточки действия, 1 подтверждение без действия (OG-5), 1 перекрёстная ссылка (OG-6)
**Метод:** минимально достаточные изменения на карточку — не переписывание сайта, устранение конкретных пробелов реестра
**Формат карточек:** идентичен `IRP_v1.md` Раздел 5 (Remediation Cards) — та же конвенция проекта, не новая

## 2. Remediation Cards

### REM-OG1 | Удаление осиротевших вкладок tab-basics / tab-narrative

| Атрибут | Значение |
|---------|---------|
| **ID** | REM-OG1 |
| **Категория** | Dead Code / Content Duplication |
| **Описание** | `tab-basics` (3 панели: Блокчейн/Ключи и адреса/Модель UTXO) и `tab-narrative` (1 панель: Ключевые события) — `display:none!important`, недостижимы через `CLUSTERS`, но всё ещё в `TAB_TO_CLUSTER` и физически в `index.html` |
| **Первопричина** | Контент перенесён в `tab-tech`/`tab-history` в прошлых итерациях, старые секции не удалены при переносе |
| **Риск** | Низкий — контент не читается пользователем (недостижим), риск не в UX, а в весе файла и потенциальной путанице при будущей навигации по коду |
| **Компоненты** | `index.html`: секции `<section id="tab-basics">`, `<section id="tab-narrative">`; объект `TAB_TO_CLUSTER` (записи `basics:`, `narrative:`) |
| **Приоритет** | P2 (не блокирует ничего, но самое дешёвое к исполнению) |
| **Сложность** | LOW |
| **Трудоёмкость** | ~30 минут (поиск всех ссылок на id + удаление + верификация) |
| **Зависимости** | Нет |
| **Стратегия** | 1. `grep -n "tab-basics\|tab-narrative"` по всему `index.html` — собрать полный список мест, не только сами секции 2. Удалить обе `<section>` целиком 3. Удалить записи `basics:'knowledge'`, `narrative:'macro'` из `TAB_TO_CLUSTER` 4. Проверить `showTab`/`updateCrumb`/любой другой код, не завязан ли он на переборе всех `tab-*` id без разбора (например, цикл по `document.querySelectorAll('.section')` — такой код не сломается от удаления, но explicit reference по id сломался бы) 5. Визуально открыть сайт, пройти по всем 14 живым вкладкам, убедиться что ничего не изменилось |
| **Ожидаемый результат** | 0 упоминаний `tab-basics`/`tab-narrative` в файле; 14 живых вкладок работают как прежде; `git diff` показывает только удаления, не изменения в живых секциях |
| **Верификация** | `grep -c "tab-basics\|tab-narrative" index.html` → 0; `node --check` на извлечённом script; полный прогон `pytest` (docs/data не затронуты, но regression-проверка дёшева) |

### REM-OG3 | Явное название третьего корня онтологии (Децентрализация)

| Атрибут | Значение |
|---------|---------|
| **ID** | REM-OG3 |
| **Категория** | Content Gap |
| **Описание** | Acc-item «Шесть свойств денег» (панель «Что такое деньги», `tab-theory`) перечисляет редкость/взаимозаменяемость/делимость/транспортабельность/прочность/верифицируемость — Децентрализация как отдельный теоретический столп BDKS Р1 не названа |
| **Первопричина** | Панель написана как классическое «6 свойств денег» (общеэкономический канон), не как прямое отражение специфичной BDKS-онтологии из 3 корней |
| **Риск** | Низкий — не искажает факты, просто не проговаривает явно центральное для Bitcoin (в отличие от золота/фиата) свойство |
| **Компоненты** | `index.html`, acc-item «Шесть свойств денег» внутри панели «Что такое деньги» (`tab-theory`) |
| **Приоритет** | P2 |
| **Сложность** | LOW — контент-правка существующего блока, без новой структуры |
| **Трудоёмкость** | ~1 час (включая формулировку текста, требующую отдельного показа пользователю перед записью — это content, не данные, но общий принцип «показать перед фиксацией» уместен и здесь) |
| **Зависимости** | Нет |
| **Стратегия** | Два варианта на выбор при реализации (не решено заранее — предложить пользователю в момент исполнения, не здесь): (а) расширить существующий пункт «Верифицируемость» одним абзацем, явно связывающим верифицируемость с отсутствием единой точки доверия (это и есть децентрализация в действии), не меняя число «шести свойств»; (б) добавить седьмой явный пункт «Децентрализация», рискуя разойтись с общеэкономическим каноном «6 свойств денег», но точнее отражая BDKS Р1. Рекомендация: вариант (а) — меньше меняет уже устоявшуюся, узнаваемую структуру панели |
| **Ожидаемый результат** | Читатель панели «Что такое деньги» видит явную связь с Децентрализацией как одним из трёх фундаментальных понятий BDKS, не только Дефицит и Верифицируемость |
| **Верификация** | Визуальная проверка текста; `node --check`; `grep` на присутствие слова «децентрализац» в нужном acc-item после правки |

### REM-OG2 | Explainer-контент для непокрытых классов онтологии (Macro, Regulation)

| Атрибут | Значение |
|---------|---------|
| **ID** | REM-OG2 |
| **Категория** | Content Gap — крупнейшая карточка плана |
| **Описание** | 6 классов BDKS Р3 без educational-панели: Regulation, Macroeconomic Environment/Central Banks, Stablecoins, Derivatives, Market Structure, Exchanges/Custodians |
| **Investigate-шаг (выполнен в рамках этого плана, не откладывается):** | Реальная частота `theme` в `signals.json` на 2026-07-06: `institutionalization`=36, `infrastructure`=12, `supply`=4, **`macro`=2, `narrative`=2**. Stablecoins/Derivatives/Market Structure/Exchanges-Custodians не являются отдельным значением `theme` вообще — 0 сигналов ссылаются на них напрямую как на тему (они могли бы упоминаться внутри `data`/`context` других сигналов, но не как классификация верхнего уровня) |
| **Вывод investigate-шага** | Приоритет по фактической активности корпуса, не по абстрактной полноте онтологии: **macro** и **narrative** (регуляторика) — единственные два класса из шести с реальным сигнальным следом (по 2 сигнала каждый, поровну — не один явно впереди другого). Остальные четыре (Stablecoins, Derivatives, Market Structure, Exchanges/Custodians) не имеют ни одного сигнала, ссылающегося на них как на тему — откладываются до появления первого такого сигнала, не создаются заранее «про запас» |
| **Первопричина** | Образовательный контент сайта исторически рос вслед за протокольной/инвестиционной частью (DCA, mNAV, сетевые эффекты, архитектура) — макро/регуляторная сторона онтологии BDKS присутствует в сигналах, но не получила своей explainer-волны |
| **Риск** | Средний — не техническая проблема, а продуктовый пробел: читатель видит сигнал с темой `macro`, но не имеет к чему привязать «что это вообще означает» на самом сайте |
| **Компоненты** | Новые панели внутри `tab-theory` (по образцу существующих 4: Что такое деньги / Семь сетевых эффектов / Стратегия DCA / mNAV) |
| **Приоритет** | P1 по содержательной важности, но выполняется после Wave 1 — самая трудоёмкая карточка, требует контент-райтинга, не только кода |
| **Сложность** | MEDIUM-HIGH — не структурная (компонент `renderTOC`/acc-item уже существует и переиспользуется), но содержательная: нужен точный, проверяемый текст по каждой теме, не шаблон |
| **Трудоёмкость** | Оценочно 3–4 часа на панель (текст + acc-items + сверка с BDKS Р3 на предмет точности формулировок) × 2 панели (macro, narrative/regulation) = 6–8 часов суммарно |
| **Зависимости** | Нет технических; есть содержательная зависимость от готовности точного текста — не начинать без отдельного шага согласования плана самой панели (черновик структуры на утверждение), по аналогии с тем, как согласовывался стиль блока СОДЕРЖАНИЕ (шрифт/цвет) в этой же сессии — контент тоже проходит через показ и подтверждение, не пишется и сразу коммitится |
| **Стратегия** | Не выполняется в рамках этого документа-плана — здесь только доведён до точки принятия решения (investigate завершён, приоритет внутри OG-2 определён: macro и narrative, не все шесть сразу). Разбить на 2 отдельные будущие сессии/PR: (1) панель «Макроэкономика и Bitcoin» — канал воздействия ставки→ликвидность→потоки капитала (BDKS Р7, диаграмма процессов уже есть в BDKS — можно адаптировать); (2) панель «Регуляторная среда» — типы регуляторных событий (BDKS Р4, таксономия событий, категория «Регуляторные»), почему асимметрична реакция (сильнее вниз при негативных, BDKS Р4). Для Stablecoins/Derivatives/Market Structure/Exchanges — не создавать панель, зафиксировать как «отложено до первого сигнала с этой темой» |
| **Ожидаемый результат** | 2 новые панели в `tab-theory` (не 6) — macro и regulation — обоснованные фактической частотой в корпусе, не абстрактной полнотой; остальные 4 класса явно помечены как отложенные с условием (появление сигнала), не забытые молча |
| **Верификация** | Отложена до фактического исполнения — эта карточка фиксирует план и приоритет, не финальный код |

### REM-OG4 | Decision Gate: 5-я категория weight (Infrastructure metrics)

| Атрибут | Значение |
|---------|---------|
| **ID** | REM-OG4 |
| **Категория** | Governance Decision — не реализуется без владельца домена |
| **Описание** | BDKS Р5 выделяет Infrastructure metrics отдельной категорией; `weight` enum CLAUDE.md — 4 значения, без неё |
| **Почему не Remediation, а Decision Gate** | Требует изменения формулы `_score_signal` (ALGORITHM.md) — веса компонентов сейчас: onchain=+4, primary=+3, market=+2, media=+1. Добавление 5-го значения требует ответа «сколько оно весит» — а это решение о ранжировании сигналов на сайте, доменное по своей природе (BAMS Финальный, Модель ответственности: «Владелец домена» отвечает за границы предметной области) |
| **Варианты для владельца домена** | (а) Добавить `weight: infrastructure`, вес по аналогии с `onchain` (тоже высокая достоверность/устойчивость по BDKS Р5) — вероятно +4 или чуть ниже (+3, между onchain и primary) — обосновать выбор явно, не просто скопировать соседнее значение; (б) явно задокументировать текущее сведение `infrastructure→onchain` как сознательное упрощение при текущем масштабе (аналогично тому, как AD-5 в NIES.md принят как «ограничение масштаба, не в очереди») — тогда OG-4 закрывается не кодом, а решением не менять |
| **Что требуется одновременно, если выбран вариант (а)** | `schemas/signal/v1.json` (`weight` enum) + `CLAUDE.md` справочник полей + `ALGORITHM.md` формула score + `docs/DATA_DICTIONARY.md` (маппинг на BDKS) — тот же класс синхронной правки, что AD-6, тем же коммитом, не постфактум |
| **Приоритет** | Не назначается — блокируется на решении, не на трудоёмкости |
| **Действие в этом плане** | Зафиксировать вопрос в форме, готовой для ответа владельца — сделано (два варианта выше). Не реализовывать ни один из двух в одностороннем порядке |

## 3. Roadmap (волны)

### Wave 1 — Дешёвые, независимые правки (сегодня, один PR или несколько мелких)

- REM-OG1 (удаление мёртвых вкладок)
- REM-OG3 (Децентрализация в тексте панели)
- REM-OG6 (перекрёстная ссылка AD-1 ↔ SBBA)

Общая трудоёмкость Wave 1: ~1.5–2 часа. Не имеет зависимостей ни друг от друга, ни от Wave 2 — можно делать в любом порядке или одним объединённым PR.

### Wave 2 — Контент (после Wave 1, отдельная сессия на согласование текста)

- REM-OG2, панель «Макроэкономика и Bitcoin»
- REM-OG2, панель «Регуляторная среда»

Не начинать без отдельного шага утверждения структуры каждой панели (черновик на подтверждение) — контент-риск выше, чем у Wave 1.

### Decision Gate (вне волн, не по расписанию — по готовности владельца домена)

- REM-OG4 — ждёт ответа на вопрос «вводить ли 5-е значение weight», сформулированный выше

---

## 4. Итог

Wave 1 закрывает 3 из 6 пунктов реестра SBBA за один короткий проход. Wave 2 закрывает содержательную часть OG-2 (2 панели из потенциальных 6 — по фактической активности корпуса, не по абстрактной полноте). OG-4 остаётся открытым вопросом до решения владельца домена — не техническим долгом, а governance-развилкой. OG-5 не требует действия (уже подтверждён как не-долг).
