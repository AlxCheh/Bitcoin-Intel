## Последовательность

Порядок определён зависимостями и тем, что не блокирует курацию сигналов —
все шесть можно вести параллельно основной работе.

### 1. Coverage Reports (#77) — первым

**Почему первым:** самая механическая задача, даёт немедленную объективную метрику
для всего, что делается дальше (включая сам Narrative Benchmarks — обе задачи об
измеримости качества).

**План:**
1. `pytest-cov>=5.0.0,<6.0.0` в `requirements.txt`
2. Шаг в `deploy.yml` job `validate`: `pytest --cov=domain --cov=infrastructure --cov=scripts --cov-report=term-missing`
3. Порог: начать non-blocking (только отчёт в лог, как `pip-audit`) — точный процент-порог
   (например, 80% для `domain/`) выставить после первого реального замера, не гадать
   заранее (тот же принцип, что PAGE_SIZE в UI Paging Strategy и лимиты в API.md —
   не финальное число до данных)
4. `.coveragerc` или секция `[tool.coverage]` в `pyproject.toml` — исключить `tests/`,
   `archive/`, `migration/v1_to_v2.py` (stub, честно 0% — это не регрессия)

**Оценка:** ~1 час, низкий риск.

### 2. SAST для собственного кода (#80) — вторым

**Почему вторым:** независима от Coverage, но использует тот же паттерн non-blocking
CI-шага — логичнее делать следом, пока контекст «добавляем новый non-blocking шаг
в validate job» свежий.

**План:**
1. `bandit>=1.7.0,<2.0.0` в `requirements.txt`
2. Шаг в `deploy.yml`: `bandit -r domain/ infrastructure/ scripts/ config/ -f screen`,
   без `-ll`/exit-on-error флагов на старте — non-blocking, по аналогии с `pip-audit`
   (`|| true` эквивалент)
3. Явно **не** сканировать `archive/`, `tests/` (тестовые моки могут триггерить
   false-positive на «hardcoded password» и т.п. — шум, не сигнал)
4. Прогнать один раз локально до коммита — если находки есть, зафиксировать
   baseline (`.bandit` skip-list с обоснованием для каждого исключения, не глушить
   молча)

**Оценка:** ~1-1.5 часа (включая разбор первых находок, если будут).

### 3. Narrative Benchmarks (#81) — третьим, честно ограниченный scope

**Почему честно ограниченный:** «оценка читаемости/связности» в полном смысле —
NLP-задача, требующая либо человеческой разметки, либо LLM-judge (которого в проде
намеренно нет, см. `docs/ERB_REPORT_v1.md` Этап 4 — детерминированный алгоритм,
не LLM-инференс). Реализовывать «квази-ИИ оценщик качества текста» противоречило бы
этому же принципу. Вместо этого — измеримые эвристики, не семантическая оценка.

**План:**
1. Расширить `tests/golden/` новым файлом `tests/golden/test_tension_benchmarks.py`
2. Эвристики (все проверяемы механически, не требуют суждения):
   - Длина `tension`: не короче ~40 символов (голый факт короче), не длиннее ~200
     (не абзац)
   - Присутствие **двух** различимых именованных механизмов/сущностей по обе стороны
     маркера («X vs Y» — X и Y должны быть разными наборами слов, не перефразом друг
     друга) — грубая эвристика через пересечение множества слов до/после маркера,
     не семантика
   - Существующая проверка формулы (заглавная буква, маркер «vs»/«несмотря на»/«при
     условии») уже есть в `quality_report.py` — **не дублировать**, только добавлять то,
     чего там нет
3. Golden-фикстуры: 5-10 примеров «хороший tension» / «плохой tension» из реальных
   таблиц ❌/✅ в `CLAUDE.md` (`ALGORITHM.md`) — уже готовый, курированный набор,
   не придумывать заново
4. **Явная оговорка в самом тесте:** это синтаксические эвристики, не оценка
   смысловой убедительности — ложные срабатывания в обе стороны возможны и ожидаемы

**Оценка:** ~2 часа. Самая рискованная по scope creep из трёх технических — держать
строго в рамках эвристик, не пытаться «допилить» до настоящей NLP-оценки в рамках
этой задачи.

## Организационные задачи — что могу подготовить, что не могу

### Definition of Ready (#76)

**Могу:** подготовить черновик раздела для `CONTRIBUTING.md` с 5 типовыми критериями
DoR, адаптированными под этот проект (например: «источник материала указан»,
«понятно к какому типу сигнала относится — см. таблицу Шаг 2 CLAUDE.md», «если это
не сигнал — сформулирована конкретная измеримая цель по образцу Acceptance Criteria
из IRP §10»).

**Не могу:** решить, какие именно критерии подходят именно вашей будущей команде —
это зависит от того, кто будет ставить задачи и в каком формате (issues, устно,
CLAUDE.md-сессия).

### RACI-матрица (#78)

**Могу:** подготовить шаблон таблицы (роль × зона ответственности: курация сигналов,
код-ревью, архитектурные решения, security-инциденты, релиз ALGORITHM_VERSION) со
столбцами R/A/C/I.

**Не могу:** заполнить фамилиями/именами — у меня нет данных о реальном составе
команды за пределами четырёх ролей, названных в IRP §14.

### Коммуникационные каналы + эскалация (#79)

**Могу:** подготовить структуру раздела в `docs/ONBOARDING.md` (где обсуждать
рутинные вопросы, куда эскалировать блокер, какой SLA на ответ).

**Не могу:** знать, какой канал реально используется (Slack? Telegram? issues?) —
это не выбор архитектуры, это факт о текущей практике команды.

**Предложение:** если скажете, какие каналы/имена/роли реально есть — заполню
черновики сразу для всех трёх; если нет — оставлю issues как есть с пометкой
«ждёт организационного решения», не буду выдумывать правдоподобные, но фиктивные
ответы.
