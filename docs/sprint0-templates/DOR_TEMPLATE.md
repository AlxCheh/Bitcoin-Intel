> Целевое место: новый раздел в `CONTRIBUTING.md`, между `## PR process` и
> `## Review checklist` (или где логичнее по факту структуры на момент
> переноса). Заполнить `[...]`, перенести, удалить этот файл.

## Definition of Ready

Задача готова к работе, если выполнено **всё** ниже:

- [ ] **Источник материала указан** — для сигнала: ссылка/скриншот/текст
      с атрибуцией; для не-сигнальной задачи: откуда взялась задача
      (issue, обсуждение, IRP/ERB-находка)
- [ ] **Тип задачи понятен** — сигнал (см. таблицу типов в CLAUDE.md Шаг 2)
      или код/документация (тогда — конкретная измеримая цель, по образцу
      Acceptance Criteria из `docs/IRP_v1.md` §10, не размытое «улучшить X»)
- [ ] **[ПЛЕЙСХОЛДЕР — критерий 3]** — например: «если задача требует
      изменения `index.html` — явно отмечено, что нужно прочитать
      `docs/spec-pilot.md» перед началом»
- [ ] **[ПЛЕЙСХОЛДЕР — критерий 4]** — например: «если задача блокирует
      другую работу — зависимость явно названа, не обнаруживается постфактум»
- [ ] **[ПЛЕЙСХОЛДЕР — критерий 5]** — например: «ответственный за задачу
      назначен (см. RACI, Issue #78), не «кто-нибудь потом возьмёт»»

## Открытые вопросы для команды

- Кто формулирует критерий 3-5 — зависит от реального процесса постановки
  задач (issues? устно? через сессию с ассистентом?), которого я не вижу
  извне репозитория
- Нужен ли отдельный DoR для «сигнальных» задач (курация) и «кодовых»
  (Sprint 0-style) — сейчас черновик объединяет оба в одном списке,
  возможно, стоит разделить
